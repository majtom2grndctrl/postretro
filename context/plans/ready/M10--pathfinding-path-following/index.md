# M10 — Pathfinding + Path Following

> **Wave:** plan 1 of 2 in the M10 closing wave (one `/orchestrate` session). Build order: **this plan → `M10--enemy-ai-behavior`**. This plan publishes the runtime steering API; the enemy-AI plan is its first real consumer. Mirrors how `M10--navigation-representation` published the region graph this plan consumes.

## Goal

Runtime pathfinding over the baked navmesh — A* across walkable regions, funnel string-pull through portal segments — plus path following that steers a movable agent through the M7 collision world toward a destination without snagging on concave walls. Completes the "movable agent" foundation and publishes the steering API the enemy AI drives.

## Scope

### In scope

- **Always-on runtime nav graph.** Build the runtime `NavGraph` whenever a loaded map carries a navmesh section, in every build — not only under `dev-tools` (today it is gated, `startup/lifecycle.rs`). The debug *overlay* stays `dev-tools`-gated; the graph and pathfinding do not.
- **Pathfinding query (pure).** A* over the region/portal graph from a start region to a goal region, then a funnel string-pull over the corridor's portal segments → an ordered list of world-space waypoints. Unreachable goal → no path. Same-region start/goal → a direct two-point path.
- **Movable agent component** — capsule geometry, velocity, grounded flag, the current path + waypoint cursor, and a destination. Engine-internal, not script-visible through `worldQuery` (the `PlayerMovement` precedent). The capsule (radius/height) is **stored per agent**, seeded at spawn from the navmesh's baked agent parameters so the agent matches what the bake eroded for. Storing it per agent (rather than reading one global size each tick) is the honest model and leaves a future per-archetype size override a descriptor field away, with no rework.
- **Agent collide-and-slide harness** — a minimal capsule sweep-and-slide (iterative slide, gravity, ground-stick) in a **new module** — build step-up and ground-stick from `collision::cast_capsule` / `cast_ray` directly — no reusable helper exists (`movement::step_up_lift` and `movement::integrate_collision` are private and player-coupled). Does not extend the 6055-line player movement module. Moves an agent along a desired horizontal velocity each tick with full world-collision response.
- **Per-tick agent steering system** — for each agent with a destination: ensure/refresh a path to it, steer toward the next waypoint, move via the harness, advance the cursor on arrival-radius (defaults to a small multiple of the agent capsule radius — not a magic constant), and set arrived / blocked (no-path) state. Replan policy: recompute when the destination moves past a threshold, when the agent strays off its corridor, or on a staleness interval; replans are capped per frame (a shared budget, time-sliced so a wave of agents cannot all replan in one tick).
- **Agent separation** — a lightweight separation term so agents converging on the same destination do not stack into one body (the M10 north star is *waves*). After path-follow velocity, push an agent away from others whose capsules overlap or fall within a separation radius. A direct pass over the agent set (entity_model §7's no-spatial-partition entity-entity approach; O(n²) is fine at wave scale). Full predictive avoidance (RVO/ORCA) and crowd-flow stay out (below).
- **Runtime steering API** (the contract plan 2 consumes) — free functions over `&mut EntityRegistry`, the `apply_damage` / `switch_animation_state` shape: `set_destination(registry, agent, pos)`, `clear_destination(registry, agent)`, a `path_state(registry, agent)` read (has-path / arrived / blocked, distance-to-destination, plus agent kinematics), a one-shot `find_path(&NavGraph, start, goal)` for callers that want a path without owning an agent, and `attach_agent(registry, entity, &NavAgentParams, move_speed)` — the public spawn seam that inserts an Agent component (capsule seeded from the passed params) on an existing entity. Plan 2 ships no agent type of its own; it consumes `attach_agent` to make an enemy movable.
- **Debug agent + path overlay** (`dev-tools`): a chord spawns a test agent that chases the player; render its corridor + funnel waypoints in-world via a new renderer-side method for the path/agent overlay following the `Renderer::emit_nav_diagnostics` wrapper precedent, keeping all wgpu renderer-side (Renderer-owns-GPU). The plan's own end-to-end demonstration; superseded by plan 2's real driver.
- **Tests** — A* (corridor correctness, unreachable, trivial same-region), funnel (straight / L-bend / single-region), agent slide (routes around a wall fixture without penetration), replan-trigger and per-frame-budget logic.

### Out of scope

- Enemy AI / FSM / attack / death, and which entity sets a destination in production — all `M10--enemy-ai-behavior` (plan 2). This plan's only destination driver is the `dev-tools` debug agent.
- Line-of-sight / visibility queries, patrol paths.
- Full predictive local avoidance (RVO / ORCA) and crowd-flow fields. A basic separation term (push apart overlapping agents) is **in** scope (above); richer avoidance is a later pass.
- Dynamic navmesh updates, dynamic obstacles.
- Off-mesh links (jump / drop-down) and region hints (cover) — future navmesh portal kinds.
- Multiple agent sizes — one baked canonical agent (the navmesh records it).
- Refactoring or unifying the player movement substrate — the agent harness is its own minimal thing.
- Path post-processing beyond funnel string-pull (spline smoothing, etc.).

## Acceptance criteria

- [ ] A* over a hand-built navmesh returns the region corridor connecting two regions; an unreachable goal returns no path; a goal in the start region returns a trivial direct path (runnable unit tests on hand-built `NavMeshSection`s).
- [ ] Funnel string-pull yields a direct two-point path through a straight corridor, bends at exactly the inner-corner portal endpoint of an L-shaped corridor (the bend waypoint's XZ coordinate equals the L's inner-corner portal endpoint — a broken-handedness straight-collapse must not satisfy this), and returns start→goal directly for a single region (runnable unit tests).
- [ ] An agent steered into a wall fixture slides along it and never ends inside the collider (Task 3 harness-only AC — runnable unit test against a `CollisionWorld` trimesh fixture — no GPU).
- [ ] An agent steered toward a point reachable only around an L-wall reaches it without penetrating the wall (full steering AC — path + funnel + slide together — runnable against a trimesh fixture with a hand-built navmesh — no GPU). The trimesh wall fixture and the hand-built navmesh corridor must be geometrically consistent; a shared helper that derives both from one wall description keeps them in agreement.
- [ ] An agent that reaches its destination reports *arrived*; an agent whose destination has no path reports *blocked* and does not walk into the wall (runnable unit test on the steering state surface).
- [ ] Two agents pathing to the same destination do not converge into overlapping positions — the separation term keeps their capsules apart (runnable unit test: agents seeded overlapping resolve to non-overlapping over several ticks).
- [ ] The steering API exposes set/clear destination, a path-state query (has-path / arrived / blocked, distance-to-destination), and a one-shot find-path operation (signature/compile-level test).
- [ ] Path replans are bounded per frame by a fixed budget — with more agents needing a replan than the budget allows, only up to the budget recompute in one tick, the rest the next; the test asserts on a per-agent replan counter (or a returned replan count from the tick function) so the bounded behavior is directly observable (runnable unit test on the budget logic).
- [ ] (a) RUNNABLE: a default-features build (`dev-tools` off) test invokes the steering API (`set_destination`, `path_state`, `find_path`) and the path query, proving they compile and run without the feature. (b) REVIEW/GREP GATE (not a runnable test): confirm the overlay (`nav_diagnostics::emit`, `Renderer::emit_nav_diagnostics`) and debug-agent spawner are wrapped in `#[cfg(feature = "dev-tools")]` attributes — inspect feature gates, do not rely on the build alone.
- [ ] With `dev-tools`, the debug agent set to chase the player routes around walls and corners on `content/dev/maps/campaign-test` without clipping, its path drawn by the overlay (review-gate: manual check; the map must first be compiled to `.prl` via `prl-build` — the `.prl` is a build artifact, not committed — and pathfinding/overlay only have data if the bake emitted a NavMesh section; verify the NavMesh section is present in the compiled `.prl` before running the check — if absent the AC is unmet, not vacuously satisfied).

## Tasks

### Task 1: Always-on runtime nav graph

Move the runtime `NavGraph` construction out of the `#[cfg(feature = "dev-tools")]` block in `startup/lifecycle.rs` so the graph is built in every build when `LevelWorld.navmesh` is present. The `nav_graph` field is `#[cfg(feature="dev-tools")]`-gated at every site today, and all must be de-gated together or the build breaks (a struct-literal cfg mismatch is a hard compile error): the field declaration, every `App` struct-literal initializer (production and test fixtures), the level-unload reset, and the load-path construction block. Grep `nav_graph` across the crate — every match is a de-gate site. Also remove the now-stale `#[allow(dead_code)]` on `mod nav` (nav is consumed in every build once pathfinding lands). Keep the navmesh debug overlay (`ToggleNavOverlay`, `nav_diagnostics::emit`) `dev-tools`-gated; the `renderer.emit_nav_diagnostics(nav_graph)` call site must keep compiling once the field is no longer an `Option`-behind-cfg. No query-surface changes — `NavGraph::region_at` / `region_portal_iter` / `regions` / `portals` / `agent_params` (`crates/postretro/src/nav.rs`) are consumed as-is.

### Task 2: Pathfinding query (A* + funnel)

A pure pathfinding query over `NavGraph` (new module under `nav/`). A* over the region graph: nodes are regions, edges are portals (`NavPortal` joining `region_a`/`region_b`), edge cost and heuristic are XZ-plane distances — portal-segment midpoints for edge cost, region centroids (midpoint of `world_min_xz` / `world_max_xz` — no centroid accessor exists) for the heuristic. Resolve start and goal regions via `region_at`; outside-all-regions → no path. Produce a region corridor, then run the Simple Stupid Funnel algorithm over the corridor's ordered portal segments (`NavPortal.left` / `right`) to emit the tightest world-space waypoint list within the corridor. Portal `left`/`right` are bake-fixed relative to `region_a < region_b`, not to traversal direction — when the corridor crosses a portal from `region_b`→`region_a`, swap left and right for the funnel. Output a path (waypoints) or none. No agent or collision dependency; unit-tested on hand-built sections.

### Task 3: Movable agent component + collide-and-slide harness

A new engine-internal agent component (new `ComponentKind` variant in `scripting/registry.rs` — the variant requires matching arms in the compiler-enforced exhaustive surfaces alongside it: the `ComponentValue` enum, `ComponentValue::kind()`, the `COUNT`/`VARIANTS` array, and a `Component` impl): capsule radius/height, velocity, grounded flag, current path + waypoint cursor, destination. Not exposed through `worldQuery`. The component constructor takes radius and height (or a `NavAgentParams`) as arguments — it does not call `NavGraph::agent_params()` internally. The `NavGraph::agent_params()` read happens at the spawn call site (the Task 5 debug spawner; the data-archetype attach in plan 2), not inside the constructor or this harness. Expose a public free fn `attach_agent(registry, entity, &NavAgentParams, move_speed)` that inserts the Agent component on an existing entity with its capsule seeded from the passed params — the single spawn seam both the Task 5 debug spawner and plan 2's data-archetype attach call (plan 2 ships no agent type of its own). When constructing the parry `Capsule` from the stored fields, note that parry expects a half-height argument (the distance from center to one endpoint, excluding the sphere), not the total height: half-height = `height / 2.0 − radius`, mirroring the player movement capsule construction. A minimal capsule collide-and-slide harness in a new module: iterative slide via `collision::cast_capsule`, step-up and ground-stick built from `collision::cast_capsule` / `cast_ray` directly — no reusable helper exists (`movement::step_up_lift` and `movement::integrate_collision` are private and player-coupled), gravity integration — takes a desired horizontal velocity + position + dt, returns resolved position + grounded. Does **not** touch `movement/mod.rs`. Unit-tested on a wall fixture.

### Task 4: Agent steering system + runtime steering API

A per-tick agent system (new module + a thin `run_agent_tick` wrapper in `main.rs`). The insertion point in `main.rs`: after the camera-follow block and before `run_weapon_fire_tick` — so agents chase the settled player and their positions settle before hitscan reads them. `nav_graph` is borrowed read-only the same way `collision_world` is, coexisting with `registry.borrow_mut()` (the `run_movement_tick` / `run_weapon_fire_tick` borrow precedent). Per agent with a destination: refresh the path (Task 2) under the replan policy + per-frame budget, steer toward `waypoints[cursor]`, move via the harness (Task 3), advance the cursor within an arrival radius (defaults to a small multiple of the agent capsule radius — not a magic constant), set arrived / blocked. Add the **separation** term: the separation push is computed against a start-of-tick snapshot of agent positions and then applied — matching the movement wrapper's snapshot pattern — so the push is order-independent (not perturbed by agents moved earlier in the same loop). After the path-follow velocity, push the agent away from other agents whose capsules overlap or fall within a separation radius — a direct pass over the `ComponentKind::Agent` set (O(n²), fine at wave scale), summed into the steering velocity before the harness sweep. The steering API (the seam plan 2 drives) is free functions over `&mut EntityRegistry`: `set_destination` / `clear_destination` / `path_state`, plus the one-shot `find_path`. Depends on Tasks 1–3.

### Task 5: Debug agent + path overlay (`dev-tools`)

A `dev-tools` chord (the `Alt+Shift+` namespace, alongside `ToggleNavOverlay`; must satisfy the `Alt+Shift+`-namespace chord test) spawns a test agent; the agent targets the player-pawn Transform when a pawn exists — use `iter_with_kind(ComponentKind::PlayerMovement)` and read its `Transform` (the pawn-discovery idiom, `entity_model.md` §7b precedent) — else falls back to the camera position. The agent sets its destination each tick (the "chase me" demo). Add a renderer-side method for the path/agent overlay following the `Renderer::emit_nav_diagnostics` wrapper precedent, keeping all wgpu renderer-side (Renderer-owns-GPU); the overlay cannot call `nav_diagnostics::emit` directly (it is `pub(super)`, render-module-private). Depends on Task 4.

## Sequencing

**Phase 1 (concurrent):** Task 1 (always-on graph), Task 2 (pure A*/funnel — takes a `NavGraph` arg), Task 3 (agent component + harness) — mutually independent.
**Phase 2 (sequential):** Task 4 — the steering system + API consumes the graph (1), the path query (2), and the agent + harness (3).
**Phase 3 (sequential):** Task 5 — the debug agent + overlay consumes the steering API (4).

## Rough sketch

- Path query: `nav/path.rs` (or extend `nav.rs`) — `find_path(&NavGraph, start: Vec3, goal: Vec3) -> Option<Vec<Vec3>>`. A* over regions; Simple Stupid Funnel (Mikko Mononen) over `NavPortal` segments. Portal "left/right" handedness is already fixed by the bake (`region_a < region_b`, sorted) — verify orientation against the corridor direction when funneling.
- Agent harness: new module (e.g. `agent/` or `nav/steering.rs`) calling `collision::cast_capsule` / `cast_ray`. Model the slide loop on `movement::integrate_collision`'s structure (iterative project-and-advance, step-up, ground-stick) but standalone and agent-shaped — copy the *pattern*, not the player-coupled code.
- Steering tick: thin `App::run_agent_tick(dt)` wrapper in `main.rs` (the `run_weapon_fire_tick` precedent), iterating `registry.iter_with_kind(ComponentKind::Agent)`; system logic lives in the new module, keeping the `main.rs` delta to the wrapper + call site.
- `NavGraph` is read-only after load; the steering system borrows it from the `App`/level alongside the `CollisionWorld`.
- Steering API: free fns in the agent module — `set_destination(&mut EntityRegistry, EntityId, Vec3)`, `clear_destination`, `path_state(&EntityRegistry, EntityId) -> AgentPathState`; one-shot `find_path(&NavGraph, Vec3, Vec3) -> Option<Vec<Vec3>>`; spawn seam `attach_agent(&mut EntityRegistry, EntityId, &NavAgentParams, f32)`. Separation: a direct O(n²) push-apart over `ComponentKind::Agent` entities, summed into the steering velocity before the harness sweep.

## Boundary inventory

None. The agent component is engine-internal (the `PlayerMovement` precedent — no script surface, no FGD KVPs, `entity_model.md` §7b). The steering API is Rust-internal, consumed by plan 2's Rust AI system. The agent variant adds an engine-internal serde tag but no wire / PRL / FFI / script surface; `NavMeshSection` already exists.

## Open questions

- **Off-navmesh agents.** If an agent or its destination resolves outside all regions (`region_at` → `None`), the agent reports *blocked*. A nearest-region snap could be kinder but risks pathing into walls; deferred unless the debug demo shows agents stranding on `campaign-test`. Consumer-resolved by plan 2 (the AI decides what to do when blocked — idle, face player).
- **Funnel granularity vs. region fragmentation.** Accepted risk — the navmesh bake already logs region count as the feedback instrument; no action in this plan.
