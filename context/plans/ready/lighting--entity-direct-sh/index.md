# Baked Static Direct SH for Entities + Billboards

> **Status:** ready (promoted after two review rounds; mechanical fixes applied, goal-derived clarifications pinned).
> **Track:** Lighting — closes the half-built dynamic-object lighting model. Follows M9 (octahedral SH GI) and rides on M10's mesh pass.
> **Related:** `context/lib/rendering_pipeline.md` §4 (lighting), §8 (shader composition), §9 (skinned model pipeline) · `context/lib/experimental_spikes.md` (Task 0 validation harness convention) · `context/plans/in-progress/M10--mesh-render-pass/` · sibling `research.md` (code-grounding citations).

## Goal

Give moving objects (mesh entities, billboards) the room's baked direct brightness they currently lack. The engine bakes lighting in two disjoint parts: lightmaps carry DIRECT light on static surfaces; the octahedral SH volume carries INDIRECT (bounced) light only. Entities and billboards can't use lightmaps, so they are lit by indirect SH alone and read much dimmer than the lightmapped geometry around them. Add a SECOND, dense, baked SH layer carrying the DIRECT light arriving at each probe from STATIC lights (per-light, shadow-ray visibility, projected into L2 SH). Dynamic shaders sample indirect + direct and finally match the room — at no runtime light-loop cost, because it is baked.

Net runtime symmetry after this lands: static surface = lightmap-direct + SH-indirect; entity/billboard = SH-direct + SH-indirect.

## Scope

### In scope

- **Projection-convention validation harness (GATING FIRST STEP).** Before the dense bake or shader work, a one-probe / one-light A/B parity harness that bakes the direct-at-probe SH for a single probe under a single static light, reconstructs irradiance at a light-aligned test normal (within ~±45° of probe→light), and asserts the per-channel relative error `abs(diff) / orig.max(1e-3)` is within the pinned threshold (ceiling 5% on-axis; off-axis asserts sign/monotonicity only). This empirically confirms (or corrects) the D3 lobe form + normalization and is the gate for all downstream work.
- **Dense direct-at-probe SH bake.** For each probe, project the direct radiance from every STATIC light (the same `StaticBakedLights` set/order the lightmap baker uses), shadow-ray-visibility-tested, into L2 SH using the existing projection + cosine-lobe convolution. Dense (all probes), no animation.
- **Per-probe light-reach culling (in v1).** Before the per-probe shadow-ray pass, cull non-reaching static lights per probe (falloff-range + portal-reachability, analogous to the delta SH bake's affinity-cell clip), with the SAME determinism guarantees. Culling skips only provably-zero contributions, so baked output stays byte-identical to an unculled bake.
- **BC6H-at-rest compression of the direct atlas (in v1).** The direct atlas is stored BC6H-compressed at rest, reusing the same encoder/format the lightmap irradiance atlas already uses, and decoded by the hardware path at load. Octahedral tile geometry is unchanged (BC6H re-encodes only the texel payload), so the sampler math is untouched.
- **A SEPARATE direct atlas.** The direct data lives in its own octahedral atlas, tile geometry IDENTICAL to the indirect atlas. ONLY the dynamic shaders sample it. The shared indirect total atlas (`sh_total_atlas`) is unchanged.
- **New PRL section.** A new section id (mirroring `OctahedralShVolume`/`DeltaShVolumes`) carries the direct atlas. No `SH_VOLUME_VERSION` bump — existing v7 maps stay loadable (they simply lack the new section).
- **Entity + billboard shaders sample indirect + direct.** `skinned_mesh.wgsl` and `billboard.wgsl` add a direct atlas sample alongside the existing indirect sample and SUM the two irradiance terms before the albedo multiply.
- **Debug toolbar control.** A direct-scale slider + an isolation toggle (direct-only / indirect-only / combined) for the dynamic path, to verify brightness parity against lightmapped surfaces. Includes wiring a scale uniform into the mesh/billboard path (which today read no such field).
- **Bake caching + determinism parity.** The direct bake participates in the SAME build-stage cache and determinism contract as the indirect bake: byte-identical output, index-derived jitter, no RNG, no order-dependent reduction, cache-keyed on geometry + static-light set + probe layout.
- **Storage footprint reporting.** The dense direct atlas roughly doubles dense SH atlas bytes; this is measured and logged.

### Out of scope (non-goals)

- **Runtime group-2 dynamic-direct light loop.** The reserved mesh-pipeline group 2 ("dynamic-direct") for genuinely moving / script-spawned lights is a SEPARATE, later feature. This spec handles ONLY the baked static direct that entities are missing. Group 2 stays unallocated.
- **Sharp per-mover shadows.** Baked direct is intentionally soft and probe-resolution — that is the correct ceiling for movers under a baked term. Crisp dynamic shadows are the dynamic-direct/shadow-map track's job.
- **Animated static lights' direct.** Animated direct already lives in `lm_anim` (the animated weight-map bake). Folding animated lights into the direct-at-probe bake would double-count — animated lights are excluded here.
- **`forward.wgsl` / static-surface lighting changes.** Static surfaces keep lightmap-direct + SH-indirect, byte-for-byte. The static path is not touched.
- **A version-bump rebake of existing content.** No `SH_VOLUME_VERSION` change; old maps load unchanged.

## Decisions

**D1 — Direct lives in a SEPARATE atlas; the indirect total is untouched (LOAD-BEARING).** `sh_total_atlas` is the composed indirect total (base + animated delta) and is sampled by BOTH `forward.wgsl` (static) and the dynamic shaders. Folding direct into it would leak direct into the static path, double-counting against the lightmap — the exact failure the architecture forbids (`sh_bake.rs:34`, `delta_sh_bake.rs:8-12`). Direct gets its own atlas, sampled ONLY by the dynamic shaders. `forward.wgsl` is not modified; it stays indirect-only by not being pointed at the new data. "Static path unchanged / no double-count" is an acceptance criterion.

**D2 — The per-frame delta compose path is NOT the vehicle.** The delta layer is a sparse CSR-by-affinity-cell structure for ANIMATED lights, scaled by per-frame time curves in `sh_compose.wgsl`. The direct layer is STATIC (no curve) and DENSE (all probes). It does not belong in the per-frame delta CSR loop. Bake a standalone dense direct atlas and sample it directly — two octahedral atlas evaluations in the dynamic fragment (indirect + direct). Composition coherence is guaranteed by the shared byte-identical probe grid regardless; hoisting the per-probe weight/validity/Chebyshev computation into a shared helper (one computation, two atlas fetches) is a PERFORMANCE optimization to avoid redundant ALU, and is the intended path — calling the full sampler twice is also correct (it recomputes identical weights), just redundant. The fragment does two 8-probe atlas reads sharing one weight pass (Task 5's preferred implementation). (Alternative considered: a compose-time pre-sum into a dynamics-only "incident" atlas. Rejected as default — see Task 7; recommend the simpler two-loop approach unless a measured reason appears.)

**D3 — Probe-point direct projection convention (CENTRAL design decision; CONFIRMED-BY-HARNESS).** A probe is a point in air — there is no surface normal at bake time. "Direct at a probe" is therefore defined as: for each reaching static light, compute its incident radiance at the probe (the `light_contribution_lambert` falloff/intensity form, receiver = probe point) × `soft_visibility` shadow factor, then `accumulate_sh_rgb` that RGB value along the light's INCIDENT DIRECTION (probe→light unit vector) as a delta/cosine lobe; after all lights, `apply_cosine_lobe_rgb` so the stored coefficients are irradiance-convolved EXACTLY as the indirect bake stores them. At runtime the existing cosine-lobe convolution + the receiver fragment's own normal produce the per-fragment response — same sampler (`sample_sh_indirect_corners_depth_aware`), same math. Rationale: this is what makes the direct term reconstruct as cosine-weighted irradiance the receiver normal can read; getting the lobe direction or the normalization wrong is exactly what breaks brightness parity with the lightmapped room. **This convention is empirically gated by the Task 0 A/B parity harness (D7), which is the source of truth: if the harness shows the lobe form or normalization is wrong, D3's exact form is corrected from the harness findings before the dense bake + shader work proceed.**

**D4 — Same static-light set/order as the lightmap baker; static-only.** The direct-at-probe bake iterates the SAME `StaticBakedLights::from_lights` set in the SAME order the lightmap and indirect bakes consume, and ONLY static (`shadow_type == StaticLightMap`, non-animated, non-dynamic) lights. This keeps direct-at-probe parity with the lightmapped room and avoids re-introducing the animated double-count (`lm_anim` already owns animated direct).

**D5 — New section id (`SectionId::DirectShVolume`), no version bump; identical tile geometry.** Add `SectionId::DirectShVolume` (struct `DirectShVolumeSection`) mirroring `OctahedralShVolume = 34` / `DeltaShVolumes = 27` (`DirectShVolume = 35`, the next free id), rather than bumping `SH_VOLUME_VERSION` (7) — a version bump hard-rejects existing v7 `.prl` files and forces a full content rebake. The direct atlas uses `DEFAULT_IRRADIANCE_TILE_DIMENSION`/`DEFAULT_IRRADIANCE_TILE_BORDER` (the indirect tile geometry) so `sh_sample.wgsl`'s sampler math is reused unchanged. A map without the section degrades to direct = 0 (entities fall back to indirect-only, today's behavior).

**D6 — Cache + determinism are LOCKED contracts.** The direct bake honors the indirect bake's determinism rules: byte-identical output for identical inputs; soft-visibility jitter index-derived (`soft_visibility_seed`-style, perturbing sampling never geometry); no RNG; order-preserving fan-out (no float `reduce`, no HashMap-iteration to assemble output). It cache-keys on the same input kinds (geometry + static-light set + probe layout) and slots into the existing warm/cold driver paths. The light-reach culling (D9) lives INSIDE this contract — it narrows which shadow rays run without changing baked bytes — and BC6H-at-rest (D8) is the same documented determinism exemption the lightmap atlas already takes (`bc6h.rs:3`).

**D7 — Projection convention is validated FIRST by a one-probe/one-light A/B parity harness (GATING; LOAD-BEARING for D3).** Before any dense bake or shader work, a gating validation harness (using the harness/spike-test pattern from `experimental_spikes.md`; this is NOT a build-to-learn spike — it is a permanent regression test) bakes the direct-at-probe SH for ONE probe under ONE static light, reconstructs irradiance at a known normal, and LOGS the per-channel relative error against the pinned equality (Task 0): the SH-reconstructed irradiance at a light-aligned normal `n` equals `radiance · max(dot(n, probe→light), 0)`, where `radiance` is recovered from the lightmap baker's `light_contribution_and_direction` (`lightmap_bake.rs:843`) + `soft_visibility` (`lightmap_bake.rs:1227`) by dividing back by their own `n_dot_l` — the lightmap baker's value is used only to cross-check the falloff/intensity/visibility magnitude, not as the direct comparison target for convolved irradiance. The gate is SCOPED to a LIGHT-ALIGNED test normal (receiver normal within ~±45° of the probe→light direction), where L2-SH truncation error is well-conditioned. The terminator and off-axis regime are EXCLUDED from the relative bound — near the terminator the true Lambert value approaches zero and SH ringing makes relative error unbounded; off-axis the harness asserts only sign/monotonicity, not a tight relative value. The relative metric uses a 1e-3 epsilon floor mirroring the engine's `BC6H_ROUNDTRIP_TOLERANCE_REL` pattern (`bc6h.rs:301`): per-channel `abs(diff) / orig.max(1e-3)`. The a-priori CEILING is 5% per channel on the light-aligned normal — this bound is never unhittable (a constant-factor normalization bug lands well past 5% even on-axis). The author confirms and pins the threshold TIGHTER from the first run, but NEVER looser than 5%. The harness is the gate for the dense bake (Task 2): it confirms — or corrects — the D3 lobe form + normalization before broad implementation. "The A/B parity harness passes within the pinned threshold" is an acceptance criterion; if it fails, D3 is updated from the harness findings (the harness is the source of truth, not D3's current wording).

**D8 — Direct atlas is BC6H-at-rest, mirroring the lightmap irradiance atlas (LOAD-BEARING for footprint; in v1).** The dense direct atlas roughly doubles dense SH atlas bytes, so it is stored BC6H-compressed at rest in v1 (not deferred), reusing the SAME encoder the lightmap irradiance atlas already uses — `bc6h::encode_bc6h_rgb_from_f32_rgba` (`bc6h.rs:71`), tagged `IRRADIANCE_FORMAT_BC6H` exactly as `CompositedAtlas::encode_section` (`lightmap_bake.rs:172`, BC6H route `:184`). Decode is the hardware `Bc6hRgbUfloat` path at load. BC6H re-encodes only the texel block payload — the octahedral TILE geometry (dim/border/atlas layout) is IDENTICAL to the indirect atlas (D5), so `sh_sample.wgsl`'s sampler math is untouched. The only new constraint is BC6H's ≥4 / 4-aligned dimension rule (`bc6h.rs:74`); the direct atlas dimension derivation must satisfy it (mirror the lightmap's power-of-two sizing or assert the alignment). A debug uncompressed bypass mirrors `uncompressed_irradiance` (`lightmap_bake.rs:103`) for A/B + determinism baselines. BC6H is the documented build-cache determinism exemption (`bc6h.rs:3`), and the encoder is itself deterministic, so the direct atlas inherits that determinism story (D6 still holds).

**D9 — Per-probe light-reach culling, mirroring the delta SH bake's affinity-cell clip (in v1; INSIDE the D6 determinism contract).** Direct adds a per-static-light shadow-ray pass at every probe; on light-dense maps this can dominate bake time. In v1 (not a follow-up) the direct bake culls non-reaching static lights per probe BEFORE the shadow-ray pass, using the SAME two-stage reach test the delta bake uses: (1) falloff-sphere AABB clip (`light_aabb` `affinity_grid.rs:274`, `cells_for_light` `affinity_grid.rs:194`) and (2) portal-reachability flood (`reachable_leaves` `affinity_grid.rs:156`), as assembled by `decompose_affinity` (`affinity_grid.rs:95`) and consumed by `bake_delta_sh_volumes` (`delta_sh_bake.rs:97`). A culled light is beyond falloff and/or not portal-reachable, so its `light_contribution_lambert` × `soft_visibility` is provably zero at that probe — culling skips only zero-contribution shadow-ray work and CANNOT change the baked coefficients. The cull set is derived purely from light geometry/origin + the static portal graph (index-derived, input-determined, no RNG); the bake fan-out stays order-preserving (no float reduce, no HashMap-iteration to assemble output), so the determinism + cache contract (D6) is unbroken. Baked output is byte-identical to an unculled bake; the cull report mirrors `log_per_light_culling` (`delta_sh_bake.rs:377`). The cull is the STRICT provably-zero falloff+portal test in BOTH warm and cold build modes — it does NOT inherit the warm indirect bake's lossy bounded-light dilation; "slots into the warm/cold driver paths" (D6) refers to CACHE plumbing only. The direct section is therefore exact regardless of build mode.

**D10 — Future group-2 dynamic-direct owns dynamic-tier lights ONLY; static-tier direct for movers is owned exclusively by this baked atlas (PINNED INVARIANT).** When the deferred group-2 dynamic-direct loop lands, it must evaluate ONLY dynamic-tier lights (`is_dynamic`) at runtime for movers — it must NOT re-evaluate the STATIC lights this baked direct atlas already covers, or a mover would receive static direct twice (baked SH + runtime loop). This is the same anti-double-count discipline the engine invariant requires (a light's contribution is never summed twice on one receiver) and mirrors the existing pins (`lm_anim` owns animated direct; the delta carries bounce only). The tier filter is the seam: this bake takes static-tier only (D4); group-2 takes dynamic-tier only. A future task implementing group-2 MUST honor this split; note it here so that work doesn't reintroduce the double-count.

## Acceptance criteria

### Visual / behavioral (eyeball in-engine, dev-tools)
- [ ] **(AC 1)** A mesh entity and a billboard placed in a directly-lit room read at a brightness that visibly MATCHES the lightmapped static geometry around them — not the dim indirect-only baseline they show today. Backed by a measurable component: using the Task 6 direct-only isolation toggle, place a mesh/billboard adjacent to a lightmapped surface of the same albedo, comparing on a LIGHT-ALIGNED receiver normal (within the Task 0 well-conditioned ±45° regime); assert the direct-only readback agrees with the neighboring lightmapped direct within the Task 0 pinned per-channel tolerance on that normal (off-axis, assert only sign/monotonicity, per Task 0). The measurable gating component is a headless/CPU readback: reconstruct the direct-atlas value at the probe nearest the test mesh and compare to the lightmap baker's direct at the same point/normal within the Task 0 tolerance, as a compiler/renderer test. The in-engine eyeball check is an ADDITIONAL qualitative check; the readback (not a screenshot) is the gating measurable. Both the readback check and the eyeball check must pass.
- [ ] **(AC 2)** An entity in a directly-lit cell is brighter than the same entity in a shadowed cell behind an occluder (baked shadow-ray visibility is present in the direct term).
- [ ] **(AC 3)** The debug toggle cycles direct-only / indirect-only / combined for the dynamic path; direct-only on a dynamic object reads as the room's direct brightness, indirect-only reads as today's dim baseline, combined sums them.

### Correctness / no-regression (test-gated where possible)
- [ ] **(AC 4)** `forward.wgsl` is unmodified and static-surface output is byte-identical to before this change (no direct double-count on lightmapped surfaces). A static surface's rendered color does not change.
- [ ] **(AC 5)** The shared indirect total atlas (`sh_total_atlas`) content is unchanged — the direct data is in a separate atlas/section.
- [ ] **(AC 6)** A map compiled WITHOUT the new section (legacy v7 `.prl`) loads and renders; dynamic objects fall back to indirect-only with no error. The loader does not reject it (no `SH_VOLUME_VERSION` change).
- [ ] **(AC 7)** Only static lights contribute to the direct atlas: a map whose only lights are animated produces an absent direct section (no section emitted, per Task 3 / Wire format); no animated light's direct double-counts against `lm_anim`.

### Projection-convention gate (Task 0)
- [ ] **(AC 8)** The one-probe/one-light A/B parity harness passes at ≥3 DISTINCT light-aligned test normals spanning the ±45° cone (not only probe→light exactly): per-channel relative error `abs(diff) / orig.max(1e-3)` is within the pinned threshold at each of those normals, which is AT MOST 5% per channel (the author pins it tighter from the first run, never looser). Off-axis / terminator normals are not subject to the relative bound — those assert only sign/monotonicity. The 1e-3 epsilon floor and 5% ceiling mirror the engine's `BC6H_ROUNDTRIP_TOLERANCE_REL` pattern (`bc6h.rs:301`) and the constant-radiance SH test precedent (`sh_bake.rs:1406`). (This gate confirms — or corrects — D3 BEFORE the dense bake and shader work; if it fails, D3 is updated from the harness findings.)

### Bake determinism / cache
- [ ] **(AC 9)** The direct bake produces byte-identical output across two runs on identical inputs (mirrors `sh_volume_bake_produces_byte_identical_output_on_repeated_runs`).
- [ ] **(AC 10)** A second compile of an unchanged map serves the direct section from the build-stage cache (cache hit, via the sibling cache entry keyed on geometry + static-light set + probe layout); changing geometry, the static-light set, or probe layout invalidates it and rebakes; the indirect bake's cache entry is untouched.
- [ ] **(AC 11)** Per-probe light-reach culling skips static lights beyond falloff / not portal-reachable BEFORE the shadow-ray pass, and the culled bake output is BYTE-IDENTICAL to an unculled bake (culling skips only provably-zero contributions, never changes baked values). The bake-time improvement (culled vs unculled) is measured and reported, mirroring `log_per_light_culling`.

### Format / footprint
- [ ] **(AC 12)** `SectionId::DirectShVolume` (struct `DirectShVolumeSection`) is registered and round-trips (encode → decode → equal); it does not reuse or collide with existing ids.
- [ ] **(AC 13)** The direct atlas is stored BC6H-compressed at rest (`IRRADIANCE_FORMAT_BC6H`, via `encode_bc6h_rgb_from_f32_rgba`) and decodes correctly at load via the hardware `Bc6hRgbUfloat` path; the octahedral tile geometry is unchanged so the sampler math is unaffected. The atlas dimensions satisfy BC6H's ≥4 / 4-aligned rule.
- [ ] **(AC 14)** The compiler logs the direct atlas byte footprint POST-BC6H-compression (and the dense pre-compression figure for comparison); the figures are reported in the PR alongside the indirect atlas size.

### Gates
- [ ] **(AC 15)** `skinned_mesh.wgsl` and `billboard.wgsl` pass naga validation.
- [ ] **(AC 16)** `cargo test` passes (workspace); `cargo clippy -- -D warnings` clean.

## Tasks

### Task 0 (GATING HARNESS): one-probe/one-light A/B projection-convention harness (`level-compiler` test)
Before any of Tasks 1–7, build a gating validation harness (using the harness/spike-test pattern from `experimental_spikes.md`; this is NOT a build-to-learn spike — it is a permanent regression test) that empirically validates the D3 projection convention. Bake the direct-at-probe SH for ONE probe under ONE static light using the D3 assembly (`light_contribution_lambert` receiver = probe, × `soft_visibility`, `accumulate_sh_rgb` along probe→light, then `apply_cosine_lobe_rgb`), reconstruct irradiance at a LIGHT-ALIGNED test normal (receiver normal within ~±45° of probe→light, where L2-SH truncation error is well-conditioned), and LOG the per-channel relative error against the pinned equality: the harness asserts that the SH-reconstructed irradiance at a light-aligned normal `n` equals `radiance · max(dot(n, probe→light), 0)`, where `radiance` is the normal-free falloff·intensity·color·visibility term — NOT the lightmap baker's cos-weighted value. The lightmap baker's `light_contribution_and_direction` (`lightmap_bake.rs:843`) + `soft_visibility` (`lightmap_bake.rs:1227`) is used ONLY as a cross-check of the falloff/intensity/visibility magnitude: divide its return value back by its own `n_dot_l` to recover `radiance`; it is NOT the direct comparison target for the convolved irradiance. The relative-error assertion is required at ≥3 DISTINCT light-aligned normals spanning the ±45° cone (not only the probe→light direction exactly), so a constant-factor or single-band normalization error cannot be absorbed by tuning to one sample. The relative metric uses a 1e-3 epsilon floor: per-channel `abs(diff) / orig.max(1e-3)`, mirroring the engine's `BC6H_ROUNDTRIP_TOLERANCE_REL` pattern (`bc6h.rs:301`). Also test off-axis / terminator normals — assert only sign/monotonicity there, NOT a tight relative bound (near the terminator the true Lambert value approaches zero and SH ringing makes relative error unbounded). The a-priori gate CEILING is 5% per channel on the light-aligned normals (a constant-factor normalization bug lands well past 5% even on-axis, so 5% is never unhittable by a correct implementation). The author pins the threshold TIGHTER from the first run, but NEVER looser than 5%. This harness is the GATE (D7): it confirms or corrects the D3 lobe form + normalization. Task 0 OWNS the minimal normal-free incident-radiance primitive (it is the first caller and the only validator): name it `incident_radiance_at_point`. It must reproduce falloff + spot-cone attenuation + directional intensity for ALL THREE `LightType` arms (Point / Spot / Directional) with only the receiver cosine removed. It is derived by factoring the shared falloff/spot-cone pieces OUT of `light_contribution_lambert` (which early-returns zero when `n_dot_l ≤ 0` and bundles the cone/directional branches) — NOT by reimplementing those pieces independently. The harness validates exactly that factor-out for all three light types; Task 2 then consumes the pinned primitive. This is the one small new function the bake adds — everything downstream is assembly. If it fails, fix D3 from the harness findings and re-run before proceeding. Keep the harness as a regression test so the convention can't silently drift. No dense bake, no shader edits until this passes.

### Task 1: New PRL section — direct octahedral atlas (`level-format`)
Add a new direct-SH section type `DirectShVolumeSection` (struct) with `SectionId` variant `DirectShVolume` in `crates/level-format/src` (a sibling to `sh_volume::OctahedralShVolumeSection`) carrying a dense per-probe octahedral irradiance atlas with tile geometry IDENTICAL to the indirect atlas (`DEFAULT_IRRADIANCE_TILE_DIMENSION`/`_BORDER`, same `irradiance_atlas_dimensions`/`tiles_per_row` derivation) plus the grid dims/origin/cell-size needed to bind the same `ShGridInfo`-shaped uniform. Register `SectionId::DirectShVolume` in `level-format/src/lib.rs` mirroring `OctahedralShVolume = 34` and `DeltaShVolumes = 27` (`DirectShVolume = 35`, the next free id after `OctahedralShVolume = 34`, no collision — AC 12 checks this; add the decode arm). Do NOT touch `SH_VOLUME_VERSION` (stays `7`). Provide `to_bytes`/`from_bytes` with a round-trip test, mirroring the existing section's encode/decode discipline. The Wire format section pins endianness/ordering. Because the runtime sampler (`sh_sample.wgsl`) is reused verbatim, the per-probe tile byte layout MUST match the indirect atlas tile exactly. The atlas payload is stored BC6H-compressed at rest (D8) with a format tag (`IRRADIANCE_FORMAT_BC6H`, plus the uncompressed-debug variant), so the section header carries the format tag exactly as the lightmap section does (`lightmap_bake.rs:177-191`); the BC6H encode itself happens at emit time (Task 3). Round-trip the format tag, and ensure the atlas dimensions satisfy BC6H's ≥4 / 4-aligned rule (`bc6h.rs:74`) — mirror the lightmap's power-of-two sizing or assert it.

### Task 2: Direct-at-probe SH bake + per-probe light-reach culling + cache integration (`level-compiler/src/sh_bake.rs`)
Add a direct-at-probe bake that, per probe, iterates the `StaticBakedLights` set (D4) and for each reaching static light: computes incident radiance at the probe — this is a NORMAL-FREE form, because a probe is a point in air with no surface normal; `light_contribution_lambert` is cited for its falloff/intensity terms ONLY, NOT its `n_dot_l` clamp (which requires a surface normal the probe does not have); the exact normal-free intensity primitive is the one Task 0 introduces and pins (the `light_contribution_lambert` falloff/intensity with `n_dot_l ≡ 1`) — multiplies by `soft_visibility` (the SAME shadow-ray routine the lightmap + indirect bakes use, `segment_clear` for the hard case), and `accumulate_sh_rgb`s that RGB along the light's incident direction (probe→light) per the D3 convention; after all lights, `apply_cosine_lobe_rgb`, then `pack_octahedral_irradiance_tile` into the Task 1 atlas. The Task 0 harness CONFIRMS the normal-free primitive before this dense loop is written — do NOT write the dense loop against an unvalidated convention. Reuse `ProbeGridLayout` so the direct probes land at byte-identical positions/validity to the indirect probes (no second grid).

**Per-probe light-reach culling (D9):** BEFORE the per-probe shadow-ray pass, cull non-reaching static lights using the SAME two-stage reach test the delta bake uses — falloff-sphere AABB clip (`light_aabb` `affinity_grid.rs:274`, `cells_for_light` `affinity_grid.rs:194`) + portal-reachability flood (`reachable_leaves` `affinity_grid.rs:156`), assembled by `decompose_affinity` (`affinity_grid.rs:95`) as `bake_delta_sh_volumes` (`delta_sh_bake.rs:97`) does. Note that `decompose_affinity` produces a per-LIGHT→cell CSR (cell-indexed by light), whereas the direct bake iterates probes and needs the per-PROBE→reaching-lights mapping (structurally inverted). Derive each probe's reaching-light set by mapping the probe to its affinity cell and reading that cell's light list from the CSR (or invert the CSR once up front before the per-probe loop); the AABB + portal-reachability test is applied per probe-cell, not per-(cell,light) as the delta bake does. A culled light is provably zero-contribution at that probe, so the baked coefficients stay BYTE-IDENTICAL to an unculled bake. ## Wire format

`DirectShVolumeSection` mirrors `OctahedralShVolumeSection` exactly except it carries direct (not indirect) coefficients and NO depth moments / animation data (direct is static and dense): little-endian; the same grid header (dims, origin, cell-size); a dense per-probe octahedral tile block in x-fastest linear probe order; per-probe tile byte layout BYTE-IDENTICAL to the indirect atlas tile (the runtime sampler is shared). The direct section omits depth moments by design — not because Chebyshev is disabled, but because `sample_sh_indirect_corners_depth_aware` reads depth moments from the EXISTING `OctahedralShVolumeSection`; that is valid because the direct probe grid is byte-identical in position to the indirect grid (Task 2: no second grid), so the indirect section's moments are the correct moments for the direct sample too. The runtime dynamic bind also binds the grid uniform (`ShGridInfo`) from the EXISTING indirect `OctahedralShVolumeSection`; the direct section's own grid header must be byte-identical to it and exists for round-trip/validation, not as a second runtime grid binding. The atlas payload is BC6H-compressed at rest (D8) with an irradiance-format tag (`IRRADIANCE_FORMAT_BC6H`, plus the uncompressed-debug variant) carried in the section header exactly as the lightmap section (`lightmap_bake.rs:177-191`) — BC6H re-encodes only the texel block payload, so the LOGICAL tile geometry above is unchanged. Empty-list / no-static-lights encoding: the section is omitted entirely (loader treats absence as direct = 0), matching how `DeltaShVolumes` is `Option` and skipped when no animated lights exist. State explicitly in the implementation which existing section the byte layout mirrors.

### Architectural questions — RESOLVED (owner, 2026-06)

1. **Principle reconciliation → RESOLVED; the over-reaching guideline was removed.** The old "direct illumination is fully dynamic / every light reaches direct through exactly one technique / they never re-weight each other" language mistook an **authoring** concern for an **engine** concern. It was revised in the context library: `index.md` §2 ("Baked over computed") and `rendering_pipeline.md` §4 now state that static-vs-dynamic is an authoring choice, and the actual engine invariant is narrower — a physical light's contribution must not be **double-counted on a given receiver** (overlapping static + dynamic must not over-brighten the same fragment); a single light MAY bake into more than one layer when those layers serve different receivers (static surfaces via the lightmap, movers via this separate baked layer). This spec already honors that invariant by construction (D1: separate atlas, `forward.wgsl` untouched, no fragment sums the same light twice). No remaining deviation.

2. **Future group-2 dynamic-direct double-count → PINNED (D10).** Settled as an invariant; see D10.
