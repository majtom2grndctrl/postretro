# postretro-lighting (CPU-math)

> Epic: `E19--render-stack-decomposition`. Splits `crate::lighting` into a CPU-math crate; the GPU pools move into the renderer crate at `E19--renderer-gpu`.

## Goal

Extract the wgpu-free lighting math (light packing, specular/influence packing) into a CPU-only crate, so `postretro-lighting`'s own tests never compile `wgpu`/`naga`, and so the script-light primitive wiring co-locates with it (Task 2). The packer consumers are renderer resource/slot modules plus one scripting system (the animated-light bridge); the crate's payoff is the warm-edit firewall, not cross-subsystem sharing.

## Scope

### In scope
- `postretro-lighting`: move the wgpu-free lighting code — from `mod.rs` (`pack_light_with_slot`, `pack_light`, `pack_lights`, `pack_lights_with_slots_into`, `patch_shadow_slots`, `patch_cube_slots`, `light_reaches_visible_cell`, `entity_occluder_eligible`, `GPU_LIGHT_SIZE`, `SHADOW_SLOT_BYTE_OFFSET`, `CUBE_SLOT_BYTE_OFFSET`), `influence.rs` (`pack_influence` **only** — the bare `LightInfluence` already sank to `postretro-render-data` per hub Decision 10; `pack_influence` imports it back), `spec_buffer.rs` (`pack_spec_lights`, `SPEC_LIGHT_SIZE`, flag consts `SPEC_LIGHT_TYPE_{POINT,SPOT,DIRECTIONAL}` + `SPEC_LIGHT_SDF_FLAG`). `lighting/mod.rs` splits rather than moves: the packing fns/consts/tests descend; a residual `lighting/mod.rs` stays in the binary declaring the staying GPU-pool modules (and `script_primitives` until Task 2).
- Carve `NO_SHADOW_SLOT` (`pub const … u32 = 0xFFFF_FFFF`) out of `spot_shadow.rs` — the moved packers reference it. `spot_shadow.rs` (staying until `E19--renderer-gpu`) re-exports it — `pub use postretro_lighting::NO_SHADOW_SLOT;` — so the existing `crate::lighting::spot_shadow::NO_SHADOW_SLOT` paths (`cube_shadow.rs`, `renderer_light_slots.rs`, `renderer_shadow_passes.rs`, `renderer_lighting.rs`, tests) keep compiling unmodified. A plain private `use` is not enough.
- `cone_frustum.rs` does **not** live here — it moves to `postretro-render-data` (`E19--render-data`). It is geometry/AABB math with the widest fan-out (model/weapon/cull/renderer), and it delegated to `compute_cull` (a GPU module bound for `postretro-renderer`), so homing it in lighting would have created a `lighting → renderer` dependency cycle via `compute_cull`. See `E19--render-data` and hub Decision 9.
- The packers' byte-layout constants travel with them (shared with WGSL).
- Re-point the packer consumers to the crate: five renderer modules (`render/mod.rs`, `render/renderer_init_resources.rs`, `render/renderer_light_slots.rs`, `render/renderer_resources.rs`, `render/renderer_lighting.rs` — the last calls `light_reaches_visible_cell` and uses `GPU_LIGHT_SIZE`/`NO_SHADOW_SLOT` in code, not just doc comments; its packer imports re-point, but its `spot_shadow::NO_SHADOW_SLOT` path stays as-is, served by the re-export), one scripting system (`scripting/systems/light_bridge.rs` — the animated-light bridge imports `GPU_LIGHT_SIZE` + `pack_light`), and two render test files (`render/tests/shader_tests.rs`, `render/tests/light_filter_tests.rs`). `render/renderer_types.rs` mentions the packers in doc comments alone (no import change); three more comment-only mentions may be touched up or left stale — `main.rs` (cites `lighting::light_reaches_visible_cell`), `fx/fog_reactions/set_fog_params.rs` (cites the `script_primitives.rs` file path), `level-compiler/src/chunk_light_list_bake.rs` (says `pack_spec_lights` lives in the `postretro` crate). The `cone_frustum`/`Aabb` consumers (weapon hit-zones, model, cull/shadow) now depend on `postretro-render-data`, not `postretro-lighting`.
- Depend on `glam`, `postretro-level-loader = { default-features = false }` (the packers use the cfg-free `LightType`/`MapLight`/`ShadowType`/`FalloffModel` types), and `postretro-render-data` (settled — `pack_influence` takes `&[postretro_render_data::influence::LightInfluence]`). **Not** `serde` — none of the three moved files use it; `script_primitives.rs` uses `serde_json`, so it travels behind the `script-ffi` gate. The `render-data` dep is driven by `influence.rs`, not cone math (cone math has moved out).
- Add an optional `script-ffi` feature gating the script-primitive wiring (Task 2), off by default. Pin the Cargo shape: `script-ffi = ["dep:postretro-scripting-core", "dep:serde_json", "dep:rquickjs", "dep:mlua", "postretro-entities/script-ffi", "postretro-foundation/script-ffi"]`, with `postretro-scripting-core`, `serde_json`, `rquickjs`, `mlua`, `postretro-entities`, `postretro-foundation` all optional deps — no VM dev-dependencies. `postretro-scripting-core` is the VM-coupled runtime per `scripting.md §12`, which is why it sits behind the gate; `entities`/`foundation` forward their own `script-ffi`. The wiring's non-test code names no VM types, but its inline test module drives both VMs directly (`rquickjs::Runtime`, `mlua::Lua`), so that module is gated `#[cfg(feature = "script-ffi")]` and `rquickjs`/`mlua` ride the feature as optional *normal* deps, not dev-deps. Default builds stay wgpu/VM-free.

### Out of scope
- The GPU pool structs `SpotShadowPool`, `CubeShadowPool`, `LightmapResources` (`spot_shadow.rs`/`cube_shadow.rs`/`lightmap.rs`) — they own wgpu and move into `postretro-renderer` at `E19--renderer-gpu`.
- `lighting/chunk_list.rs` (`ChunkGrid`, `CHUNK_GRID_UNIFORM_SIZE`) — CPU byte-payload planning for renderer buffers, untouched by this spec. It descends to `postretro-render-cpu` per hub Decision 11, not to lighting.
- The marshalling substrate `script_primitives.rs` calls — stays in `postretro-scripting-core` (the VM-coupled runtime per §12; lighting's gated deps are what pull the VM crates). Only the lighting *wiring* descends here (Task 2).

## Acceptance criteria
Inherits the epic global acceptance criteria — see `E19--render-stack-decomposition/index.md`. Durable decisions are captured into `context/lib/` per spec as each spec is approved — not in one batch at first promotion.
- [ ] Crate is a workspace member; `cargo build --workspace` + `cargo test --workspace` pass; light-packing tests pass from their relocated home. The descended light/sequenced/cross-VM tests are `#[cfg(feature = "script-ffi")]`-gated, so a bare `cargo test -p postretro-lighting` (default features) skips them — they only run via `cargo test --workspace` or `cargo test -p postretro-lighting --features script-ffi`.
- [ ] `cargo tree -p postretro-lighting` (default features) shows no wgpu/winit/glyphon/kira and no `mlua`/`rquickjs` — the `script_primitives` wiring, and the VM crates it pulls, are gated behind the off-by-default `script-ffi` feature, so nothing VM-coupled reaches the default graph. The feature-ON path is exercised by the workspace build itself: the binary enables `postretro-lighting/script-ffi` unconditionally, so the first AC compiles the wiring and `Session::build` still registers the light primitives.
- [ ] The re-pointed packer consumers compile against the crate: `render/mod.rs`, `render/renderer_init_resources.rs`, `render/renderer_light_slots.rs`, `render/renderer_resources.rs`, `render/renderer_lighting.rs`, `scripting/systems/light_bridge.rs`, `render/tests/shader_tests.rs`, `render/tests/light_filter_tests.rs` — some reach the packers through `render/mod.rs`'s imports via `use super::*`, so a correct implementation may not diff every file. `GPU_LIGHT_SIZE`/`SPEC_LIGHT_SIZE`/slot byte-offset constants unchanged (no shader-layout drift). (`weapon`/`model`/`cone_frustum` consumers depend on `postretro-render-data` now, not this crate.)
- [ ] The three GPU pool modules (`spot_shadow.rs`, `cube_shadow.rs`, `lightmap.rs`) still compile in their current home (they move at `E19--renderer-gpu`); the only *functional* edit among them is `spot_shadow.rs` swapping its `NO_SHADOW_SLOT` definition for the `pub use` re-export — it also needs a doc-only fix, since its intra-doc link `[crate::lighting::entity_occluder_eligible]` dangles once that fn moves out of `crate::lighting`. `cube_shadow.rs` and `renderer_shadow_passes.rs` stay untouched; every `spot_shadow::NO_SHADOW_SLOT` *reference* in the re-pointed files (`renderer_light_slots.rs`, `renderer_lighting.rs`, tests) stays as-is — only those files' packer imports change.
- [ ] `scripting.md §12` (already updated) still matches the shipped shape when Task 2 lands — confirm, don't re-edit. §12 already records that once a subsystem is its own crate, its script-primitive wiring co-locates there behind `script-ffi`, the crate-descent this spec establishes.

## Tasks

### Task 1: Extract postretro-lighting CPU-math
Create the crate. Split `lighting/mod.rs`: everything except the module declarations descends — the packing fns/consts, the `light_reaches_visible_cell` and `entity_occluder_eligible` predicates, and the inline `#[cfg(test)]` tests; the residual module root stays in the binary, declaring the staying GPU-pool modules, `chunk_list`, and `script_primitives` until Task 2. Move `influence.rs` and `spec_buffer.rs` whole. Carve `NO_SHADOW_SLOT` (`spot_shadow.rs` keeps a `pub use`). The moved packer bodies (`pack_light_with_slot`, `pack_light`) and the ~15 inline `#[cfg(test)]` references in `mod.rs` currently spelled `crate::lighting::spot_shadow::NO_SHADOW_SLOT` need rewriting to the crate-local path, not left pointing at the `spot_shadow` re-export. Widen boundary symbols — rule: every moved item with a cross-crate consumer becomes `pub`, including `NO_SHADOW_SLOT`. Re-point the packer consumers — AC 3 names the files. `cone_frustum.rs` is **not** part of this move — it lands in `postretro-render-data` (`E19--render-data`), and its consumers re-point there.

### Task 2: Descend script_primitives wiring behind `script-ffi`
**Decision (was open question 1): the scripting wiring descends into `postretro-lighting` behind the optional `script-ffi` feature.** Move `lighting/script_primitives.rs` into the crate under that feature gate, and re-point the three sites that reach it by the old path: `scripting/primitives/light.rs` (today's `pub(crate) use crate::lighting::script_primitives::*;`), `scripting/entity_world_primitives.rs` (today's `use crate::lighting::script_primitives as light;`), and `bin/gen_script_types.rs` (today's `#[path = "../lighting/script_primitives.rs"]`) — each parenthetical is the pre-move state being re-pointed, not the target. The marshalling substrate it calls stays in `postretro-scripting-core` (`primitives_registry::{ContextScope, PrimitiveRegistry}`, `sequence::{SequenceError, SequencedPrimitiveRegistry}`).

The whole-file move carries more than the light wiring: the file's `register_shared_types` (the light-module complement to the one in `scripting/primitives/mod.rs`) also registers the cross-domain query types — `WorldQueryComponent`, `WorldQueryFilter`, `Entity`, `EmitterEntity` — so those descend into `postretro-lighting` alongside the light ones. This is acknowledged debt, not an oversight: the registry emits typedefs in registration order (load-bearing for snapshot stability — `primitives_registry.rs`), so splitting the shared registrations back out would reorder emission and risk breaking the committed drift test. A future domain split is a separate, deliberate refactor that rewrites the typedef snapshot on purpose; it must account for `LightEntity` registering after `EmitterEntity` here, so the boundary cannot be drawn by naive extraction. Leave `LightComponent`'s registration as-is: `animatedSlot` is serde-visible on the struct but intentionally omitted from the typedef — do not "complete" it during the move.

One reach-back must be resolved at move time: the file's inline `#[cfg(test)]` module's `registry_for` helper calls `crate::scripting::entity_world_primitives::register_world_primitives` — `pub(crate)` in the binary and staying there, so it is unreachable from `postretro-lighting`. In practice the test module splits by dependency, not by name: whichever tests actually call `register_world_primitives` — `primitive_context_scopes` and most `world_query_*` tests — relocate into the binary alongside the staying wiring; the light/sequenced/cross-VM tests descend with a `registry_for` rewritten to register only the light primitives. Three `world_query_*`-named tests (`world_query_returns_all_light_bearing_entities`, `world_query_tag_filter_narrows_result`, `world_query_empty_when_no_match`) call `collect_light_handles` directly and descend with the crate despite the name. Both halves need their own copies of the shared test helpers (`test_ctx_with_light`, `install_all`, `install_all_lua`) and a `registry_for` scoped to what each half needs.

Registrars stay invoked as today, just re-homed: `register_sequenced_light_primitives` from `Session::build` directly, and `register_light_entity_primitives` from `register_all` (which `Session::build` calls) — both call sites stay in the binary, above the crate, preserving the one-way dependency the `script-ffi` gate relies on. The same widening as Task 1 applies here: `register_shared_types`, `register_light_entity_primitives`, `register_sequenced_light_primitives`, `collect_light_handles`, `handles_to_json`, and `LightQueryHandle` all become `pub` (the last because it's the return type of the two `pub` handle functions). Plumbing: the module itself is public under the gate — `#[cfg(feature = "script-ffi")] pub mod script_primitives;` — because two of the re-pointed sites need the whole module surface (the glob re-export in `scripting/primitives/light.rs`, the module alias in `entity_world_primitives.rs`), not just the two registrar fns. The binary depends on `postretro-lighting` with `features = ["script-ffi"]` unconditionally — no runtime-side forwarding is possible: once the gated dep exists, `postretro-scripting-core` sits *below* `postretro-lighting`, so only the binary can flip the flag. That unconditional dep also covers the `gen_script_types` bin target, which drops its `#[path]` include (and the surrounding `mod lighting { … }` shim) and inherits the re-pointed `postretro_lighting::script_primitives` imports through its included modules (`entity_world_primitives.rs`, `primitives/light.rs`), rather than importing it directly itself (the typedef drift test pins the output byte-identical per the global ACs). Cargo shape: `script-ffi = ["dep:postretro-scripting-core", "dep:serde_json", "dep:rquickjs", "dep:mlua", "postretro-entities/script-ffi", "postretro-foundation/script-ffi"]`, those six crates all optional deps and no VM dev-dependencies; the moved test module is gated `#[cfg(feature = "script-ffi")]`, so `rquickjs`/`mlua` ride the feature as optional normal deps.

Principle: this **extends** `scripting.md §12` — §12 places handler *wiring* in the binary ("not new crates"), and co-locating that wiring with its subsystem once the subsystem is itself a crate is the natural generalization. §12 directly governs the two halves this keeps: the marshalling substrate stays in `postretro-scripting-core`, and `script-ffi` is off by default per the §12 Cargo pattern. With the feature off the crate stays VM-free; with it on it pulls the VM crates. The hub names `postretro-lighting` the first instance and the precedent the Epic 16 combat crate mirrors. Confirming §12 still matches the shipped tree is a checkbox AC above.
