# Scripting-Core Extraction + Primitive-Handler Relocation

> **Status:** in-progress — implementation complete; boundary docs captured before landing
> **Related:** `context/plans/done/engine-data-floor/` (precondition — the floor this builds on, now shipped) · `context/lib/scripting.md` §12 · `context/lib/build_pipeline.md` · `context/lib/development_guide.md` · `context/plans/drafts/compile-time-reduction/` · sibling `research.md`

## Goal

Extract the VM-coupled scripting runtime — the `rquickjs`/`mlua` bindings, the marshalling orchestration, the IR store scope, and the typedef generator — out of the `postretro` binary into a `scripting-core` crate stacked on the `engine-data-floor` crates, so routine engine edits stop recompiling the extracted VM substrate. Then co-locate script-callable primitive handlers — both the pure logic and their `register_*` registration wiring — with the subsystems they expose. Cuts warm incremental edit-loop time now and reduces agent fan-out conflict ahead of the script-led Epic 16 (Combat). It does not remove scripting VM deps from `postretro`, prove cold-build wins, or slim the final binary; those are follow-ups. No runtime behavior, wire-format, or scripting-semantics change.

## Precondition

This spec **depends on `engine-data-floor`, which has shipped** (`context/plans/done/engine-data-floor/`). That plan extracted the two VM-free crates this one stacks on:

- **`postretro-foundation`** (lower) — IR evaluator core, the movement/IR cluster, foundation-clean descriptor & value PODs, sunk subsystem PODs.
- **`postretro-entities`** (upper) — entity registry, `ComponentValue`, components, `ScriptCtx`, the registries, `DataRegistry`, the entities-bound descriptors.

Both crates already carry their own types' FFI marshalling impls behind an optional `script-ffi` feature (off by default; `postretro-entities/script-ffi` forwards to `postretro-foundation/script-ffi`). The floor's earlier draft had this spec extract a single `entity-core` crate; that is **superseded** — the floor replaces it. This spec turns the floor's `script-ffi` features on and pulls the VM-coupled remainder that still lives in `postretro` after the floor lands.

## Scope

### In scope

- Extract `postretro-scripting-core`: the VM-coupled runtime that still lives in `postretro` after the floor lands — the primitive registry, the two VM subsystems, the marshalling-orchestration bridges, the IR store scope, the VM/UI converters, and the typedef generator. It is the crate that turns the floor's `script-ffi` features on.
- Rewrite the relocated code's floor-type imports from `postretro`'s `crate::scripting::*` compatibility barrels to direct floor-crate imports (`postretro_foundation::*` / `postretro_entities::*`). The floor left ~8 thin barrels in `crates/postretro/src/scripting/` (`registry.rs`, `ctx.rs`, `slot_table.rs`, `data_registry.rs`, `error.rs`, `engine_state_catalog.rs`, `components/mod.rs`, `ir/mod.rs`, plus the `data_descriptors/types/` and `data_descriptors/{error,validate}.rs` sub-barrels) that re-export floor types; ~276 scripting-internal call sites still import through them. The code that moves into `scripting-core` cannot keep those barrel paths (`scripting-core` has no `crate::scripting` tree), so its imports flip to the floor crates directly. The barrels themselves stay in `postretro` (see Decisions).
- Introduce a `ScriptingCore` sub-struct on `Session` grouping the scripting-tranche fields.
- Relocate cross-runtime tests by what they reference (a test lives where its referents live): registrar-dependent ones (calling a relocated `register_all`/`register_*`) stay in `postretro` — traveling with their handler, or as integration tests calling postretro's `register_all` + scripting-core's generator/runtime; only substrate-only ones move to `scripting-core/tests/` (or an integration crate as fallback). Keep pure-logic handler tests with the handler.
- Relocate script-callable primitive handlers (`scripting/primitives/`, `scripting/reactions/`) to co-locate with the subsystem each exposes (into subsystem module trees within `postretro`, **not** new crates), registering against `scripting-core`. **The `primitives/*` handlers are not fully runtime-agnostic** — `entity.rs` (`NullableString`), `world.rs` (`WorldQueryFilterInput`, `JsonValue`), and `store.rs` (`StoreSchemaJson`, `ScriptSlotValue`, etc.) carry handler-local marshalling newtypes with `FromJs`/`FromLua` for inbound arg newtypes, `IntoJs`/`IntoLua` for outbound result newtypes (directional, not round-trip) — `light.rs` and `mod.rs` carry **none** (light marshals floor types directly; `mod.rs` is pure registration); only the closure *bodies* are VM-free (verified: the newtypes appear in closure signatures, not bodies — bodies delegate to VM-free free functions). So `primitives/*` relocation splits: the pure handler logic **and** the `register_*` wiring that references them both relocate to the subsystem module in `postretro`; only the marshalling newtypes, their FFI impls, and the registry machinery stay in `scripting-core`, referenced by the relocated wiring as a legal down-edge. The `reactions/*` handlers are genuinely agnostic (zero `rquickjs`/`mlua` refs in **non-test** code) and relocate whole.
- Quote before/after `--timings` from the compile-time-reduction Task 1 baseline on the extraction.

### Out of scope

- The floor crates themselves (`postretro-foundation`, `postretro-entities`) and the entity/value-type FFI impls they own — shipped by `engine-data-floor`. Do not duplicate the floor's marshalling impls here; `scripting-core` only enables their `script-ffi` feature.
- Crate-ifying the bridges (`scripting/systems/`) — they stay in `postretro` (render/FX-coupled glue).
- Crate-ifying any subsystem that receives relocated handler logic (render/audio/lighting/movement, plus health/particles/mesh/command-queue per the Task 5 mapping) — handler "co-location" means same module tree in `postretro` depending on the new crates; the handler-side firewall fully lands when those subsystems crate-ify later.
- `inventory`/`linkme` distributed registration — keep explicit `register_all` → `register_*` aggregation from `Session::build`.
- Any change to primitive names, wire shapes, marshalling results, SDK typedef output, or PRL format.
- PRL-loader / visibility extraction (separate compile-time-reduction tasks).
- Removing scripting VM dependencies from the `postretro` binary, cold-build optimization, or final binary slimming. This plan only establishes an incremental edit-loop firewall.

### Task 2: Extract `postretro-scripting-core`

After the floor lands, the VM-coupled remainder still living in `postretro` is exactly what becomes the `scripting-core` crate. Per `engine-data-floor`, that remainder is:

- `ir/scopes.rs` (`StoreScope`)
- `conv.rs` — the json-orchestration bridges (`json_to_js`/`js_to_json`/`json_to_lua`/`lua_to_json`)
- `data_descriptors/mod.rs` inline `mod js { ... }` / `mod lua { ... }` blocks (declared at lines 56–74 inside `data_descriptors/mod.rs`, with no per-dir `mod.rs`) + the same file's VM/`render::ui` glob — relocating them moves those inline blocks and the parent glob, not standalone module files. The sibling `mod types { ... }` block (lines 50–54: `types::{combat, manifest, movement}`, the descriptor type structs/enums) and its `types::*` re-exports (lines 90–92) **stay** in `postretro` — descriptor concern, not VM-coupled; do **not** relocate the whole `data_descriptors/` directory or the `types` block on a "this file imports `mlua`, move it" reading. Only the `js`/`lua` blocks, their `js::*`/`lua::*` globs, and the `render::ui` glob relocate
- `RegisteredUiTree`/`LevelManifest` (in `data_descriptors/runtime_manifest.rs`)
- the `data_descriptors/validate/runtime.rs` `mlua` + `render::ui` validators (`validate_dense_lua_array`, `parse_*`) — the runtime-side validators that stayed in `postretro` post-floor; `validate/foundation.rs` descended to `postretro-foundation` and does **not** move
- `data_descriptors/vm_adapters.rs` `js_err`/`lua_err` — the VM adapters; `data_descriptors/error.rs` is now a thin barrel re-exporting `postretro-foundation`'s `DescriptorError`, so it does **not** move (the descended `DescriptorError` lives in `postretro-foundation`)
- `runtime/*`
- `luau.rs`, `quickjs.rs`, `primitives_registry.rs`, `reaction_dispatch.rs`, and the typedef generator — the registry TYPES (`PrimitiveRegistry`, `ReactionPrimitiveRegistry`, `SystemReactionRegistry`), `ReactionPrimitiveRegistry::dispatch` + the system-command drain, and the registration machinery (the sealed `RegisterablePrimitive` trait, `impl_registerable!`, the installer aliases). The per-domain `register_*` FUNCTIONS that *populate* these registries do **not** stay here — they relocate to `postretro` in Task 5, co-located with their handler logic.
- `sequence.rs` — the `SequencedPrimitiveRegistry` TYPE + `SequenceError` (its `register_sequenced_*` registrar fns relocate in Task 5)

Move this cluster into `postretro-scripting-core`. Depend on both floor crates with `script-ffi` enabled — `postretro-entities = { workspace = true, features = ["script-ffi"] }`, which forwards to `postretro-foundation/script-ffi`. This crate is the one that turns the floor's `script-ffi` on; the entity/value-type FFI **impls already live in the floor crates** under `script-ffi` (do not duplicate them). `scripting-core` owns the json-orchestration bridges, the marshalling newtypes + their FFI impls, the registry types + machinery, and the VM subsystems only — the per-domain `register_*` registrar functions that populate the registries are **not** part of this substrate move; they relocate to `postretro` in Task 5.

**The `gen-script-types` bin seam.** The typedef generator moves to `scripting-core`, but the registry it reads is populated by `register_all`, which moves to `postretro` (Task 5). The `gen_script_types` bin (`crates/postretro/src/bin/gen_script_types.rs`) stays in `postretro`: it calls the `postretro`-resident `register_all` to populate a `PrimitiveRegistry`, then hands that populated registry to `scripting-core`'s `write_type_definitions` — a legal `postretro → scripting-core` down-edge. Its current `#[path = "../scripting"]` inline-include of the soon-to-be-`scripting-core` modules (`conv`, `primitives_registry`, `typedef`, `sequence`) converts to real `postretro_scripting_core::*` imports, mirroring the floor's earlier migration of the bin off `#[path]` re-includes for the floor types. The debug-build path is parallel: `Session::build` calls `emit_sdk_types_in_debug` the same way.

Only the **generator code** moves down. The `typedef/` generator's `register_all`-consuming TEST modules — `typedef/tests/committed.rs` (the byte-identity drift test) and `typedef/tests/mod.rs`, plus the sibling `typedef/tests/{surface,snapshots}.rs` that also call `register_all` — do **not** travel with the generator into `scripting-core`. They reference the relocated `register_all` (a `postretro` symbol under (c)), so they stay in `postretro` as integration tests calling postretro's `register_all` + scripting-core's generator — the same legal `postretro → scripting-core` down-edge as the `gen-script-types` bin.

The move includes rewriting the relocated code's floor-type imports: each `crate::scripting::*` reference to a floor type (routed today through `postretro`'s thin compatibility barrels — `registry.rs`, `ctx.rs`, `slot_table.rs`, `error.rs`, etc.) becomes a direct `postretro_foundation::*` / `postretro_entities::*` import. This is forced, not stylistic — `scripting-core` has no `crate::scripting` tree for the barrel paths to resolve against, and routing through `postretro`'s barrels would make `scripting-core` depend on the `postretro` binary, a circular edge since `scripting-core` sits below it. It mirrors how the floor already flipped external/gameplay consumers (movement/nav/weapon/netcode) to direct floor imports. The barrels stay in `postretro` for the scripting-adjacent code that stays there (the `scripting/systems/*` bridges, `Session`, and the relocated `register_*` wiring).

Default deps beyond the floor crates: `rquickjs`, `mlua`, `serde`, `serde_json`. Keep the sealed `RegisterablePrimitive` trait + `impl_registerable!` macro and the installer type aliases here. Note: the scripting tree that becomes `scripting-core` contains no `unsafe` today (grep over `crates/postretro/src/scripting/` for `unsafe fn|impl|{|trait` returns nothing), and `postretro-foundation` denies `unsafe` crate-wide (`#![deny(unsafe_code)]`), so the "No `unsafe` added" AC is satisfied by there being none to inherit.

### Task 3: `ScriptingCore` sub-struct on `Session`

Behavior-preserving.

**Exact field set that moves into `ScriptingCore`** (verified against `Session` in `crates/postretro/src/session/mod.rs`):

- The script runtime and context: `script_runtime: ScriptRuntime`, `script_ctx: ScriptCtx`.
- The persisted registries the `Session::build` aggregation populates: `sequence_registry: SequencedPrimitiveRegistry`, `reaction_registry: ReactionPrimitiveRegistry`, `system_registry: SystemReactionRegistry`.
- The `ScriptCtx`-clone consumers constructed from `script_ctx.clone()` at the build site: `player_hud_state` (`PlayerHudStatePublisher`), `flash_decay` (`FlashDecay`), `vignette_decay` (`VignetteDecay`), `shake_decay` (`ShakeDecay`), `input_mode_tracker` (`InputModeTracker`). (`presentation_cells` is a presentation-only `PresentationCellStore::new()` that takes **no** `ScriptCtx` — group it with the tranche or leave it on `Session` per implementation taste; it is not load-bearing to the firewall either way.)

**`classname_dispatch` disposition — stays a direct `Session` field, OUT of `ScriptingCore`.** Source-confirmed: `classname_dispatch: ClassnameDispatch` is engine-owned built-in-classname dispatch (`ClassnameDispatch::new()` populated by `register_builtin_classnames(&mut classname_dispatch)` at `Session::build`); it takes **no** `ScriptCtx`, is not a `ScriptCtx`-clone consumer, and is not one of the four script registries. It maps `classname` strings to engine spawn handlers and survives level unload as engine state. It is therefore **not** part of the scripting tranche — leave it as a direct `Session` field.

### Task 4: Relocate cross-runtime tests

Core principle: **a test lives where the things it references live.** A test that calls a relocated registrar (`register_all` or any per-family `register_*`) cannot live in `scripting-core/tests/` — that would name a `postretro` symbol under (c), the forbidden backwards edge. Split accordingly:

- **Registrar-dependent tests stay in `postretro`** — two sub-cases:
  - **Handler-resident parity / cross-runtime tests travel with their handler** into the `postretro` subsystem module as `#[cfg(test)]` next to the relocated handler (in Task 5, Phase 4). They may instantiate `rquickjs::Runtime` + `mlua::Lua` because `postretro` names the VM types under (c). Example: the `#[cfg(test)]` block in `set_fog_params.rs`.
  - **The typedef drift test and other `register_all`-consuming tests** (`typedef/tests/committed.rs`, `typedef/tests/mod.rs`, and the `register_all` callers in `luau.rs`/`quickjs.rs`/`runtime/`) stay in `postretro` as integration tests, calling postretro's `register_all` + scripting-core's generator/runtime — the legal `postretro → scripting-core` down-edge, the same shape as the `gen-script-types` bin.
- **Only substrate-only cross-runtime tests move to `scripting-core/tests/`** — those that exercise `scripting-core`'s own converters / registry machinery / VM runtime **without** a postretro registrar. Where they need registered entries, build a minimal registry inline (the repo already has a `mini_registry` pattern in `typedef/tests/mod.rs`). A standalone `scripting-integration` crate remains the fallback only per Decisions.
- **Keep pure-logic handler tests** (e.g. `apply_light_animation_inner`-style) with their pure-logic in the `postretro` subsystem module, so subsystem `cargo test` needs no VM.

Handler-resident VM-touching test modules relocate to the `postretro` subsystem **with their handler** in Task 5 (Phase 4), not here — this avoids a backwards dependency where Task 4 would move tests out of files Task 5 hasn't relocated yet.

### Task 5: Relocate primitive + reaction handlers

Relocate each handler to co-locate with the subsystem it exposes. **There is no single `register_all` reaction surface** — handlers register through **four distinct registry types**, all constructed at the single `Session::build` site (`crates/postretro/src/session/mod.rs`). A handler relocates *with all of its registration sites*. The `register_*` wiring + the pure handler logic both relocate to subsystem modules in `postretro`; only the marshalling newtypes + their FFI impls and the registry types/machinery stay in `scripting-core` (the VM crate), referenced by the relocated wiring as a legal down-edge.

| Registry | Registrar(s) (current location) | Takes `ScriptCtx`? | Handler shape |
|---|---|---|---|
| `PrimitiveRegistry` | `register_all` → `register_*` per domain (`primitives/mod.rs`) | yes | closure capturing `ScriptCtx` |
| `SequencedPrimitiveRegistry` | `register_sequenced_fog_primitives` (`reactions/registry.rs:161`); `register_sequenced_light_primitives` (`primitives/light.rs:197`) — travels with the light primitive relocation, not the reactions relocation | yes | `(id, args)` closure capturing `ScriptCtx` |
| `ReactionPrimitiveRegistry` | `register_emitter_reaction_primitives`, `register_fog_reaction_primitives` (`reactions/registry.rs`) | **no** | `(reg, targets, args)` closure — calls the per-handler module `dispatch(reg, targets, &parsed)` (e.g. `set_emitter_rate::dispatch`, called at `registry.rs:82` after deserializing `args → parsed`); distinct from `ReactionPrimitiveRegistry::dispatch(&self, name, registry, targets, args)` (`reactions/registry.rs:54`), which is the registry's own dispatch method |
| `SystemReactionRegistry` | `register_system_reaction_primitives` (`reactions/system_commands.rs`) | **no** | enqueues onto `SystemCommandQueue` |

All four registrar **functions** relocate to `postretro`, co-located with the handler logic they wire (the "current location" column above is a factual pre-move record — it is what relocates). `scripting-core` retains the registry **types**, the marshalling newtypes + FFI impls, the converters, and dispatch. `SystemCommandQueue` itself lives in `postretro-entities` (the floor); the **drain** runs through `reaction_dispatch.rs` (in `scripting-core`) via the `ScriptCtx::system_commands` field. Several families register in **more than one** registry — **fog** registers in both `ReactionPrimitiveRegistry` and `SequencedPrimitiveRegistry`; **light** registers in both `PrimitiveRegistry` (via `register_all`) and `SequencedPrimitiveRegistry`. Move every site for a family together.

**Two relocation shapes (per A1, scope):**

- **`reactions/*` (agnostic, zero non-test VM refs):** relocate the whole handler — its `Args` type, `dispatch` fn, and the registrar closure — to its subsystem module. The `dispatch(reg, targets, &parsed)` signature already takes native args (no VM types). `system_commands` handlers enqueue onto `SystemCommandQueue` (in `postretro-entities`, from the floor); the **drain** runs through `reaction_dispatch.rs` via the `ScriptCtx::system_commands` field and stays in `scripting-core` (not the `scripting/systems/` bridges). Confirm zero `rquickjs`/`mlua` import in moved non-test code; any VM-touching `#[cfg(test)]` module goes to `scripting-core/tests/` (Task 4).
- **`primitives/*` (the wiring references marshalling newtypes):** split **in place** — intra-file surgery on `primitives/{light,world,store,entity,mod}.rs`, **not** whole-file moves. The marshalling newtypes + their FFI impls stay in `scripting-core`; the registrar fns + per-primitive closures + the pure-logic free fns move to the `postretro` subsystem module. (Do not move whole `primitives/*.rs` files — each file is partitioned.) **Both** the pure handler logic **and** the `register_*` wiring (the registrar fn + its per-primitive closures) relocate to the subsystem module in `postretro`. What stays in `scripting-core` is only the marshalling newtypes and their `FromJs`/`IntoJs`/`FromLua`/`IntoLua` impls. The relocated wiring references those newtypes — a legal down-edge (`postretro` → `scripting-core`) — and may transitively name the installer aliases (`QuickJsInstaller`/`LuauInstaller`); the relocated pure-logic must not import `rquickjs`/`mlua`. Confirm the pure-logic fn is VM-free; the wiring keeps a thin VM-type surface at the registration boundary.

  - `world.rs`: `WorldQueryFilterInput` (`FromJs` + `FromLua`) — the inbound `worldQuery` filter arg; `JsonValue` (`IntoJs` + `IntoLua`) — outbound serde-shaped result wrap.
  - `store.rs`: `StoreSchemaJson` (`FromJs` + `FromLua`); `ScriptSlotValue` (`FromJs` + `FromLua`); `Any` (`IntoJs` + `IntoLua`); `StoreStateRefs` (`IntoJs` + `IntoLua`); `StoreDefinition` (`IntoJs` + `IntoLua`); `StoreDeclarationManifest` (`IntoJs` + `IntoLua`). (Note: `store.rs` also carries VM-free pure-logic free fns — `define_store`, `read_store_slot`, `write_store_slot`, `write_script_store_slot`, `apply_store_slot_batch`, `write_state_slot_json`, `apply_text_edit`, `store_declaration`, the validators — and the `drain_store_declarations_{js,lua}` drains; the free fns relocate to `postretro`, the drains carry VM types and stay with the newtypes in `scripting-core`.)
  - `light.rs`: **none** — the closures marshal floor types directly (`Option<LightAnimation>`, `LightComponent`, `Vec3Lit`); there is no handler-local newtype to leave behind. The whole light primitive surface (pure-logic `apply_light_animation`/`apply_light_animation_inner`/`validate_and_normalize`/`collect_light_handles`/`handles_to_json`, the `register_light_entity_primitives`/`register_sequenced_light_primitives` registrars, and `register_shared_types`) relocates to the `postretro` lighting subsystem.
  - `mod.rs`: **none** — it carries only `register_all`, `register_shared_types`, and the submodule wiring; all relocate to `postretro`.

  **The seam:** the relocated `register_*` closure (now in `postretro`) does the marshalling — decode script args via `scripting-core`'s newtypes, then call the **same-crate** pure-logic fn with native Rust args plus the `ScriptCtx`/cluster handle, then encode the result via `scripting-core`'s newtypes. There is **no backwards edge**, because both the closure and the pure-logic fn live in `postretro`; only the newtypes they reference live below in `scripting-core`. The pure-logic fn signature takes native types only — never `rquickjs`/`mlua` types. This already matches source: closure bodies today are one-line delegations to free fns (`apply_light_animation`, `read_store_slot`, `parse_query_filter` + collectors), so the split relocates those free fns **and** the closures that wrap them together. For `worldQuery` specifically, `parse_query_filter` + its collectors **and** the `register_*` closure (with the `WorldQueryFilterInput`→native unpack and the `JsonValue` result wrap) all relocate to the `postretro` entity subsystem; `scripting-core` owns only the `WorldQueryFilterInput` / `JsonValue` newtypes + their FFI impls. (This resolves the open boundary-type question: the native intermediate that crosses the seam is produced and consumed entirely within `postretro`.)

`Session::build` remains the sole aggregation site: all four registries are constructed there. The registrar fns now live in `postretro` subsystem modules and are *called* from this single site (the firewall holds — these are `postretro`-side calls into co-located wiring, not edges into `scripting-core`). Registration order must be preserved across the move — now the order is the `register_*` call sequence at the `Session::build` aggregation site plus the `gen-script-types` bin.

For `PrimitiveRegistry`, the canonical reference is `primitives/mod.rs`'s current `register_all`, whose call sequence is **exactly** (verbatim, in this order):

```
register_shared_types(registry);
light::register_shared_types(registry);
light::register_light_entity_primitives(registry, ctx.clone());
store::register_store_primitives(registry, ctx.clone());
world::register_world_primitives(registry, ctx.clone());
entity::register_entity_primitives(registry, ctx);
```

This is the AC-7 byte-identity contract: when the per-domain registrars are re-invoked from `Session::build` after relocation, this sequence must be reproduced verbatim (`register_shared_types` → light shared types → light primitives → store → world → entity). Order is an invariant since the typedef generator's byte-identical output (AC) depends on it: `iter_types()` iterates registered types in registration order (load-bearing per `primitives_registry.rs` doc comment), while `visible_primitives()` sorts primitives alphabetically by name before emission. Both orders must be preserved.

**Re-export nit:** when `register_all` relocates, a `crate::scripting::primitives::register_all` re-export path must survive for the many postretro-internal `#[cfg(test)]` and bin callers that import it through that path today, OR every such call site is updated to the new module path. Grep-confirmed callers include `bin/gen_script_types.rs`, `main.rs`, `startup/lifecycle.rs`, `session/mod.rs`, `render/ui/lifecycle_render_test.rs`, `scripting/quickjs.rs`, `scripting/luau.rs`, the `scripting/runtime/tests/`, `scripting/ir/{parity,e2e}_tests.rs`, and the `scripting/typedef/tests/` modules. Pick one (preserve the re-export, or sweep the call sites) — do not leave them dangling.

## Sequencing

**Precondition:** `engine-data-floor` has shipped — `postretro-foundation` and `postretro-entities` exist with their `script-ffi` features in place. This spec does not start until then.

**Phase 1 (sequential):** Task 1 — baseline gates all timing claims.
**Phase 2 (sequential):** Task 2 — `scripting-core` extraction pulls the VM-coupled remainder onto the floor and enables `script-ffi`. The crate must exist before the sub-struct and test/handler relocations can target it.
**Phase 3 (concurrent):** Task 3, Task 4 — independent once the crate exists (Session grouping vs. cross-runtime test relocation). Task 4 moves only tests not tied to a handler that Task 5 will relocate; Task 4's relocated tests instantiate raw `rquickjs::Runtime` + `mlua::Lua` (not `Session`), so they don't touch Task 3's `ScriptingCore` restructuring.
**Phase 4 (sequential, after Task 3):** Task 5 — per-handler relocations fan out across handler families (each is an independent unit), but the phase as a whole runs after Task 3 because both Task 3 and Task 5 edit `Session::build`'s registration block (`register_all` call site + clone distribution). Within the phase, handler families parallelize. Handler-resident parity tests (e.g. `set_fog_params.rs`'s `#[cfg(test)]` block) travel **with their handler into the `postretro` subsystem module** (as `#[cfg(test)]` next to the relocated handler) here in Phase 4 — they call a relocated registrar, so under (c) they stay in `postretro`, not `scripting-core/tests/`. This removes the backwards dependency that would arise if Task 4 moved them in Phase 3.

Structural surgery (the extraction in Phase 2) is serialized deliberately; the handler fan-out is Phase 4, gated behind Task 3's `Session::build` restructuring. The shape is "serialize structural surgery, fan out handler relocation."

## Rough sketch

**Crate layout.**
- Floor (from `engine-data-floor`, precondition): `crates/foundation/` (`postretro-foundation`) and `crates/entities/` (`postretro-entities`). Both VM-free by default; each owns its types' FFI impls under `script-ffi`.
- `crates/scripting-core/` (`postretro-scripting-core`): the VM-coupled runtime. Depends on `postretro-entities = { workspace = true, features = ["script-ffi"] }` (which forwards to `postretro-foundation/script-ffi`), plus `rquickjs`, `mlua`. This is the crate that turns `script-ffi` on.

**The firewall.** Cargo feature unification compiles the floor crates once with `script-ffi` on in the full `postretro` build, but their default crate builds stay VM-free. `scripting-core` is where the runtime VM substrate lives. Editing a relocated handler or bridge in `postretro` should rebuild only `postretro`, not `postretro-scripting-core` or VM `-sys` crates. This is the achieved edit-loop firewall. It is not a claim that `postretro` is VM-free, that cold builds avoid VM work, or that release binaries are smaller.

**Aggregation site unchanged.** `register_all` (now a thin aggregator) and its sibling registrars for the other three registries (`register_emitter_reaction_primitives`/`register_fog_reaction_primitives`, `register_system_reaction_primitives`, `register_sequenced_light_primitives`/`register_sequenced_fog_primitives`) now live in `postretro`, co-located with their handler logic; they stay explicit (no `inventory`/`linkme`) and are still invoked from the single `Session::build` site (in `postretro`). `scripting-core` provides the registry types + registration machinery they populate. The "one indivisible group" stays indivisible at runtime (shared `Rc`) and becomes the `ScriptingCore` type at compile time.

**Naming.** Working name `postretro-scripting-core`; final name decided at implementation.

## Landing Boundary

Before landing, treat this as the final boundary:

- **`postretro-scripting-core` owns:** VM runtimes, runtime construction, registry types and installer machinery, reaction dispatch, sequenced registry, VM converters/adapters, primitive marshalling newtypes, store bridge helpers, descriptor/UI VM conversion, SDK typedef generation, hot-reload watcher substrate.
- **`postretro` owns:** `Session` / `ScriptingCore` construction, primitive aggregation, subsystem handler logic and registrar wiring, scripting systems bridges, built-in classname dispatch, state persistence, and compatibility barrels.
- **Compatibility barrels intentionally remain:** `crate::scripting::{conv,data_descriptors,game_state_refs,ir_scopes,luau,luau_require,luau_virtual_modules,primitives_registry,quickjs,reaction_dispatch,refresh_plan,runtime,sequence,staged_manifest,watcher}` over `postretro-scripting-core`; floor barrels such as `ctx`, `registry`, `slot_table`, `data_registry`, `error`, `engine_state_catalog`, `components`, `ir`; primitive barrels under `crate::scripting::primitives::{light,store,world,entity}`; reaction barrels under `crate::scripting::reactions::{mod,registry,system_commands}`.
- **Import rule:** new engine code imports core-owned scripting APIs from `postretro_scripting_core` directly. It imports floor-owned types from `postretro_foundation` / `postretro_entities` directly. Do not add new `crate::scripting::*` compatibility-barrel imports except in existing compatibility wrappers and tests that still exercise legacy paths.
- **Observed firewall check:** after touching `crates/postretro/src/lighting/script_primitives.rs`, `cargo check -p postretro --timings` completed in 1.70s and rebuilt only `postretro`; `postretro-scripting-core`, `rquickjs-sys`, `mlua-sys`, and `luau0-src` appeared in `target/cargo-timings/cargo-timing-20260629T154430106Z-892a9f914c1db502.html` only as zero-duration graph entries.
- **Deferred:** removing direct VM deps from `postretro`, deleting compatibility barrels, crate-ifying renderer/audio/input/FX/gameplay subsystems, PRL-loader or visibility extraction, cold-build optimization, and final binary-size work.

## Decisions (resolved during review)

- **The floor is a precondition, not a task.** `engine-data-floor` ships `postretro-foundation` + `postretro-entities` (and their `script-ffi` impls). The earlier single-`entity-core` extraction is superseded; this spec stacks `scripting-core` on top.
- **Cross-runtime tests split by what they reference (a test lives where its referents live).** Forced by the (c) registrar placement: a test that calls a relocated registrar (`register_all` / `register_*`) cannot live in `scripting-core/tests/` (that names a `postretro` symbol = the forbidden backwards edge). So registrar-dependent tests stay in `postretro` — traveling with their handler as `#[cfg(test)]`, or as integration tests calling postretro's `register_all` + scripting-core's generator/runtime (the down-edge). Only **substrate-only** cross-runtime tests (no postretro registrar) move to `scripting-core/tests/`. The original fallback clause — "a standalone `scripting-integration` crate is the fallback only if a test drags handler-specific fixtures that don't belong in `scripting-core`" — already anticipated handler-specific tests not belonging in `scripting-core`; under (c) that becomes the **primary** path for registrar-dependent tests (they stay in `postretro` rather than spilling into a fallback crate), and the standalone crate remains a fallback for substrate-only tests that still drag fixtures.
- **A1 (primitive-handler marshalling):** only the `primitives/*` marshalling newtypes + their FFI impls stay in `scripting-core` (the VM substrate). **Both** the pure handler logic **and** the `register_*` wiring relocate to the `postretro` subsystem — because the wiring *calls* the logic, and `scripting-core` sits below `postretro` in the one-way crate dependency, so it may never depend up on `postretro` (see Scope, Task 5, and the registrar-placement decision below).
- **Registrar placement — wiring co-locates with handler logic in `postretro`.** The `register_*` registrar functions and their per-primitive closures relocate to `postretro`, not `scripting-core`. Forced by the one-way crate dependency (`foundation → entities → scripting-core → postretro`) that is the firewall's basis: the wiring calls the subsystem pure-logic, and §12's handler-co-location contract puts that logic in the binary, so `scripting-core` — sitting below — may never name it. Sealed by this spec's goal of cutting multi-site fan-out conflict ahead of the script-heavy Epic 16 (Combat): under this resolution a new primitive is a **one-crate** change in `postretro` (logic + wiring together), versus the rejected fn-pointer-injection alternative's **three-site** change (scripting-core wiring + postretro logic + a `Session::build` injection seam). Accepted cost: `postretro` keeps a thin VM-type surface at the registration boundary (the wiring references `scripting-core`'s marshalling newtypes and may transitively name the installer aliases). This surface is acceptable because the extracted VM substrate is firewalled in `scripting-core`; it does not make `postretro` VM-free.
- **Barrel fate — relocated code imports the floor crates directly; the barrels stay in `postretro`.** The floor left ~8 thin `crate::scripting::*` compatibility barrels (`registry.rs`, `ctx.rs`, `slot_table.rs`, `data_registry.rs`, `error.rs`, `engine_state_catalog.rs`, `components/mod.rs`, `ir/mod.rs`, plus `data_descriptors/types/` and `data_descriptors/{error,validate}.rs` sub-barrels) re-exporting floor types, with ~276 scripting-internal call sites still routed through them. Code moving into `scripting-core` rewrites those barrel imports to direct `postretro_foundation::*` / `postretro_entities::*` imports. This is **forced by the dependency direction, not a preference**: `scripting-core` has no `crate::scripting` tree for the barrel paths to resolve against, and routing through `postretro`'s barrels would invert the dependency (`scripting-core` sits below `postretro`). The barrels are **not** removed — they remain in `postretro` to serve existing scripting-adjacent code, tests, and compatibility wrappers. New engine code should not add imports through them for core-owned or floor-owned APIs. This matches the pattern the floor already used to flip external/gameplay consumers (movement/nav/weapon/netcode) to direct floor imports.
- **Phase 4 stays in this spec — not split out.** Task 5's handler relocation is ~8 independent, low-conflict families across subsystems — the shape `/orchestrate` fans out well, so handler count is not a reason to split. Keeping it unified maximizes progress in one orchestration session and avoids re-opening `Session::build` in a later spec (Task 3 and Task 5 both touch its registration block; doing them in one spec touches the file once). The firewall (Phases 1–3) and the co-location (Phase 4) land together. The Phase 1–3 boundary remains independently shippable if implementation later forces a stop, but that is a fallback, not the plan.
