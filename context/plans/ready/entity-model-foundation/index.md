# Entity Model Foundation

## Goal

Complete the Milestone 6 entity layer. The registry, component structs, and scripting primitives are already built. What's missing is the data pipeline connecting `.map` entity definitions to the runtime: a PRL entity section so `world.map_entities` is populated at level load; an expansion of script-defined archetypes to carry component descriptors; and two reference behavior scripts that validate the full stack end-to-end.

## Context

The registry (`scripting/registry.rs`) is done: `EntityId`, `EntityRegistry`, all component kinds, tags. Scripting primitives are done: `spawnEntity`, `despawnEntity`, `entityExists`, `getComponent`, `setComponent`, `emitEvent`, `sendEvent`, `registerHandler`. The classname dispatch table (`scripting/builtins/mod.rs`) is wired in main.rs and handles `billboard_emitter`. The `DataRegistry` and `LevelManifest` deserialization are done. The event system (`registerHandler` for `levelLoad`/`tick`) is done.

The three concrete gaps:
- `world.map_entities` is always empty — the PRL format has no generic entity section; the compiler discards non-light entities.
- `EntityTypeDescriptor` carries classname only; data scripts cannot attach components to script-defined archetypes.
- `worldQuery` only resolves `"light"` to a component kind; other component strings are rejected at the FFI boundary.

## Scope

### In scope

- PRL map-entity section: compiler, format crate, runtime parser
- `EntityTypeDescriptor` expansion to carry optional component values
- `registerEntity` mod-scope primitive accepting component descriptors
- Data-archetype spawn path at level load: map entities matched to data-registry archetypes
- `worldQuery` expansion to all `ComponentKind` variants
- `game_events` ring buffer: `emitEvent` appends a `GameEvent` entry to a capped ring buffer owned by the engine. **Storage:** a new `game_events: Rc<RefCell<VecDeque<GameEvent>>>` field on `ScriptCtx` (`scripting/ctx.rs`), distinct from the existing `events: Rc<RefCell<EventQueue>>` on the same struct. **This is additive to, not a replacement for, `ctx.events.broadcast`** — the broadcast queue (`VecDeque<ScriptEvent>` drained each frame by `event_dispatch`) remains the per-frame dispatch path. The new ring buffer is a separate observability log: `emitEvent` pushes to both (broadcast for dispatch, ring buffer for the log). **Capacity:** 1024 entries. **Eviction policy:** when full, pop oldest (`pop_front`) before pushing the new entry. **`GameEvent` struct fields** (defined alongside `ScriptEvent` in `scripting/ctx.rs`): `kind: String` (event name, mirrors `ScriptEvent.kind`), `frame: u64` (engine frame counter at emit time; sourced via a new `frame: Rc<Cell<u64>>` field added to `ScriptCtx` alongside `registry`/`events`/`handlers`, incremented by `main.rs` once at the start of the Game logic phase (before any game logic runs, including `fire_tick`) — `emitEvent` reads it via `ctx.frame.get()` when constructing the `GameEvent`), `payload: serde_json::Value` (mirrors `ScriptEvent.payload`). Today the buffer is drained to `tracing::info!(target: "game_events", ...)` at the end of the Game logic phase, immediately after `self.script_runtime.fire_tick(...)` returns in `main.rs` (the same call site that drives `event_dispatch`); each log line includes the `kind`, `frame`, and JSON-serialized `payload` fields of the `GameEvent`. Emit via `tracing::info!(target: "game_events", kind = %ev.kind, frame = ev.frame, payload = %ev.payload)` — `serde_json::Value`'s `Display` impl produces canonical JSON output, so `%` (Display) is the correct sigil here, not `?` (Debug). A future UI layer reads the buffer directly. Seeded here as the observability layer for `DamageSource`; intended long-term as a CRPG-style game log for modders.
- Reference behaviors: `RotatorDriver` and `DamageSource` as TypeScript/Luau scripts
- Entity API coverage added to `docs/scripting-reference.md`

### Out of scope

- Parent/child transform hierarchy (grounded movement)
- Velocity integration, physics, collision (grounded movement)
- BSP leaf tracking per entity
- New `ComponentKind` variants
- New FGD entity types beyond the two reference behaviors
- Player entity
- Hot reload of data scripts (already documented as engine-restart–only)

## Acceptance criteria

- [ ] A `.map` with a non-light, non-worldspawn entity compiles to a `.prl` that loads without error. `world.map_entities` is non-empty. Unknown classnames are logged at debug level and skipped; the engine does not crash.
- [ ] A mod-scope script calls `registerEntity` with an emitter component descriptor. At level load, map entities with the matching classname are spawned with an emitter component attached. `worldQuery({ component: 'emitter' })` returns them.
- [ ] A mod-scope script calls `registerEntity` with a light component descriptor. At level load, entities spawn with a light component attached. `worldQuery({ component: "light" })` returns script-defined lights alongside map-authored ones.
- [ ] `worldQuery` accepts every component kind string (`"transform"`, `"light"`, `"emitter"`, `"particle"`, `"sprite_visual"`). Unknown component strings return a `ScriptError`.
- [ ] `spawnEntity` called with a tags array produces an entity whose `worldQuery({ tag: ... })` resolves it by the supplied tag.
- [ ] `worldQuery({ component: "particle" })` returns an empty list without error. (Particles are fully managed by Rust per `entity_model.md` §8 — "scripts never observe individual particles" — so the string is whitelisted to avoid `ScriptError` on a known component name, but the result is intentionally empty. The same applies to `"sprite_visual"`.)
- [ ] A mod-scope script that calls `registerEntity({ classname: "...", components: { ... } })` produces an entity-type entry in the Rust data registry whose component descriptors are present and non-empty (e.g. the emitter or light component descriptor is attached). The `components` field is **not** silently stripped at the SDK boundary or at deserialization.
- [ ] A behavior script can read a per-placement key-value pair authored on a `.map` entity (e.g. `"myKey" "myValue"`) for an entity spawned via the data-archetype path. The verifiable observable is: a script-side read returns `"myValue"` for `"myKey"` and a sentinel (null / nil / absent) for an unset key. The accessor's exact name and call shape is settled in Task 1's KVP-read subtask.
- [ ] Given a classname registered both as a built-in classname and via `registerEntity`, only the built-in spawn path runs. A warning is logged identifying the conflict.
- [ ] When a test map contains multiple entity instances sharing a conflicting classname, the conflict warning logs exactly once for that classname — not once per instance.
- [ ] The `RotatorDriver` script is loaded in a test map. The `tick` handler fires at the fixed tick rate. An entity with the `game_rotator_driver` classname advances orientation each tick.
- [ ] The `DamageSource` script's `levelLoad` handler runs at level load and unconditionally logs at `debug!` level either the count of resolved target entities found or an explicit "no targets" line. The handler's execution is observable from logs even when the test map contains zero matching entities.
- [ ] The `DamageSource` script's `emitEvent("damage", ...)` call, fired automatically every 3 seconds via the `tick` handler, completes without throwing a script exception. An entry appears in the `game_events` log (`RUST_LOG=game_events=info`). The script remains alive afterward — subsequent `tick`-handler invocations continue to fire. This verifies the engine's event dispatch correctly no-ops when no handler is registered for the `"damage"` event name rather than crashing the script VM.
- [ ] `cargo test --workspace` passes.
- [ ] `cargo test --workspace` includes the existing type-definition drift test; it passes after regenerating `sdk/types/postretro.d.ts` and `sdk/types/postretro.d.luau` to reflect all Task 2 surface changes: (a) the expanded `worldQuery` filter set, (b) snake_case `ComponentValue` kind tags, (c) the new `registerEntity` mod-context primitive, (d) the updated `spawnEntity(transform, tags?: string[])` signature, (e) the flat `ComponentValue` shape replacing `{ kind, value }`, and (f) the KVP-read accessor signature (once path is chosen in Task 1).
- [ ] `docs/scripting-reference.md` covers: `spawnEntity`, `despawnEntity`, `entityExists`, `worldQuery`, `getComponent`, `setComponent`, `emitEvent`, `sendEvent`, and the `registerEntity` mod-context primitive.

## Tasks

### Task 1: PRL map-entity section

**Compiler** (`postretro-level-compiler`): after resolving brush entities and lights, collect remaining non-worldspawn, non-light `.map` entity entries and write a new `MapEntity` PRL section. Each entry: `classname` (string), `origin` (Vec3), `angles` (Vec3), tags (`Vec<String>`), remaining key-value pairs as a list of (key, value) string pairs. The compiler parses the `_tags` KVP (space-delimited, matching the existing `LightTags` convention) into the tags list and strips it from the generic KVP bag — behavior scripts never see `_tags` as a raw key. The full set of reserved keys stripped before the generic KVP bag is written is: `classname`, `origin`, `_tags`, `angle`, `angles`, and `mangle` — these surface as structural fields on every `MapEntity` (classname, origin, engine-convention angles, tags), so stripping them prevents scripts from observing duplicate or conflicting data via the KVP-read accessor. `_tags` is parsed exactly once per entity by this new collection pass; brush-entity and light flows run before this pass and use separate paths — no double-parse occurs. The `_tags` convention follows Quake's underscore-prefix reserved KVP pattern (`_color`, `_fade`); map authors set per-instance tags on individual placements in TrenchBroom. The `angles` field is converted from Quake `pitch yaw roll` (degrees) to engine-convention Euler angles at compile time. This conversion belongs in the Quake format adapter layer (`level-compiler/src/format/quake_map.rs`) alongside the existing axis swizzle and unit scale — not in shared compiler logic. PRL is the engine's internal coordinate standard; the `format/` layer is the adapter boundary. A future format (UDMF, Blender export) would supply its own adapter and produce the same engine-convention output. `parse_mangle_direction` in `quake_map.rs` is the existing precedent for this conversion. Scripts see engine convention only and are never exposed to Quake angles.

**Format** (`postretro-level-format`): add a new section type for the entity list. Wire format follows the existing section-table pattern. Assign section ID 29 (see PRL section inventory in `build_pipeline.md` for the full list).

**Runtime** (`postretro/src/prl.rs`): parse the new section into `world.map_entities` (`Vec<MapEntity>`). The `MapEntity` struct already exists in `scripting/builtins/mod.rs`. It currently has fields `classname: String`, `origin: Vec3`, `key_values: HashMap<String, String>`, and `tags: Vec<String>`, plus a `diagnostic_origin()` helper. This task adds `angles: Vec3` (engine-convention Euler angles); `classname`, `origin`, `key_values`, and `tags` are preserved as-is. As part of this task, move it to `scripting/map_entity.rs` (a new module) and re-export it from both `scripting/mod.rs` and `scripting/builtins/mod.rs` so existing call sites compile unchanged. The parser in `prl.rs` and the dispatcher in `scripting/builtins/mod.rs` then both depend on this neutral module rather than `prl.rs` reaching into a dispatcher-owned type. The level load path in `main.rs` already calls `apply_classname_dispatch` on this slice; once populated, built-in entities (e.g. `billboard_emitter` placed from TrenchBroom) spawn automatically.

**KVP read accessor.** The current `getComponent` primitive only returns the five `ComponentKind` variants — it cannot expose a `MapEntity`'s `key_values: HashMap<String, String>`, and `MapEntity` itself is consumed during dispatch and not retained on the spawned ECS entity. The level-load sweep therefore borrows `world.map_entities` — `apply_classname_dispatch` takes `&[MapEntity]`, not ownership. Both the built-in dispatch pass and the data-archetype sweep (Task 2) iterate the same slice; neither consumes it. The caller in `main.rs` retains ownership and may drop the slice after both passes complete. To make per-placement KVPs visible to behavior scripts, the data-archetype spawn path (Task 2) must persist the source `MapEntity`'s `key_values` onto the spawned entity in a form a primitive can read. Two paths are acceptable; pick one during implementation:

1. Persist `key_values` into a side-table keyed by `EntityId` on the registry, and add a primitive (`getEntityProperty(id, key) -> string | null` (TypeScript) / `string?` (Luau) — matching the `Option`-to-`null` convention used by other engine-FFI returns) that reads from it. This is the smallest surface and stays out of `ComponentValue`. Built-in classname handlers should also write into this table for entities they spawn so KVP access is uniform regardless of spawn path.
2. Add the KVP map as fields on a new component variant. This is a heavier change and crosses the "no new `ComponentKind` variants" out-of-scope line; if chosen, expand the out-of-scope list and justify in a note.

The verifiable observable (covered by the AC above) is symmetric across both paths: a script-side read returns the authored value for a known key and a sentinel for an unset key. The Rust-side primitive's signature lands in `sdk/types/postretro.d.ts` and the type-definition drift test catches stale exports.

If implementation reveals that path 1 also conflicts with another scope boundary, the alternative is to push KVP read access out of this plan and into a follow-up; in that case, remove the KVP-access AC and add a Note here explaining what blocks it. Do not silently leave the KVP pipeline compiled-in but unobservable.

### Task 2: Script archetype expansion and `worldQuery`

**`EntityTypeDescriptor` expansion** (`scripting/data_descriptors.rs`): add optional component fields — `light: Option<LightDescriptor>` and `emitter: Option<BillboardEmitterComponent>`. `LightComponent` cannot serve as a descriptor: it carries bake-pipeline fields (`falloff_model`, `cast_shadows`, cone configuration, `origin`, `light_type`) that are not author-settable from script-init time, and `conv.rs` explicitly blocks constructing a `ComponentValue::Light` from JS/Lua via `setComponent`. The `setComponent` rejection arm for `Light` is renamed to `light` as part of the casing sweep; the rejection behavior is preserved. Note: the Rust enum variant `ComponentValue::Light` stays PascalCase — only the serde wire tag and the string-keyed match arms (kind string `"Light"` → `"light"`) change. The new descriptor path is independent: `registerEntity` reads a `LightDescriptor`, never a `ComponentValue::Light`. Define `LightDescriptor` alongside `EntityTypeDescriptor` in `data_descriptors.rs` with only the authorable fields: `color: [f32; 3]`, `intensity: f32`, `range: f32` (mapped onto `falloff_range` at spawn), `is_dynamic: bool`. JS/Lua-facing field names use snake_case to match `BillboardEmitterComponent` (which is `#[serde(rename_all = "snake_case")]`) and the snake_case sweep applied to `ComponentValue` below — so the field crosses the FFI as `is_dynamic`, not `isDynamic`. `LightDescriptor` itself carries `#[serde(rename_all = "snake_case")]`. The pre-existing camelCase on `LightComponent` is internal storage only; descriptors crossing through `registerEntity` are a new surface and follow the unified snake_case convention. The data-archetype spawn path constructs a full `LightComponent` from the descriptor by filling bake/runtime fields with defaults (or values driven by per-placement KVPs from the source `MapEntity`). Spawn defaults for the non-descriptor fields on `LightComponent` (`scripting/components/light.rs`): `origin = MapEntity.origin` (cast to `[f32; 3]`); `light_type = LightKind::Point` (`Spot` and `Directional` are not author-settable from `registerEntity` — point-light is the only shape descriptor-spawned lights take in this plan); `falloff_model = FalloffKind::InverseSquared`; `cone_angle_inner = None`, `cone_angle_outer = None`, `cone_direction = None` (cone fields unused for point lights); `cast_shadows = false`; `animation = None`. The descriptor's `range` populates `falloff_range`, and `is_dynamic` is copied directly, subject to the dynamic-only override described in the data-archetype spawn path below (descriptor-spawned lights are forced dynamic regardless of the descriptor's `is_dynamic` value). Update `entity_descriptor_from_js` and `entity_descriptor_from_lua` to read the optional `components` sub-object and parse `light` / `emitter` fields into the new optional fields. Add deserialization tests covering: descriptor with `components.emitter` only; descriptor with `components.light` only; descriptor with both; descriptor with neither.

**KVP override semantics.** Per-field overridability declared on the descriptor type is out of scope for this plan — the proposed `Option<LightDescriptor>` / `Option<BillboardEmitterComponent>` are concrete structs with no mechanism to mark individual fields as overridable, and adding one would require a richer descriptor type that doesn't fit the current scope. The simpler rule used here: any `initial_`-prefixed KVP key (e.g. `initial_rate`, `initial_spread`, `initial_range` for `LightDescriptor.range`, `initial_is_dynamic`) present on a map placement unconditionally wins over the descriptor's default value at spawn time; the spawn path parses each `initial_`-prefixed KVP value against the target field's type — scalar fields (`f32`, `bool`, `u32`) parse via `FromStr`, and `[f32; 3]` fields parse as three space-delimited `f32`s (matching the existing `_color`/`_tags` whitespace convention) (for `initial_color`, values are linear `[0.0, 1.0]` floats — no normalization is applied; the same convention used by `_color` on FGD light entities). On parse failure the spawn path warns with the entity's diagnostic origin, the offending key name, and the raw value, then keeps the descriptor's default for that field; load continues. There is no generic reflection mechanism — the spawn path hand-writes a per-field match on the `initial_`-prefixed key string for each field of `BillboardEmitterComponent` and `LightDescriptor`. Override keys use snake_case: the literal prefix `initial_` plus the descriptor's Rust field name (e.g. `initial_rate` for `rate`, `initial_spread` for `spread`, `initial_range` for `LightDescriptor.range`, `initial_is_dynamic` for `is_dynamic`). The override key is derived from the descriptor field name (`range`), not the spawned component's field name (`falloff_range`). This matches the existing FGD KVP convention, where keys like `spin_rate`, `initial_velocity_x`, `cast_shadows`, and `data_script` are uniformly snake_case. The `initial` prefix is a convention signaling to map authors that the value is a spawn-time default, mutable by behavior scripts at runtime. A future iteration may add per-field overridability declarations on a richer descriptor type. Two fields remain unconditionally immutable: `id` (never carried as a KVP) and component kind. Entity type is fixed at spawn — to "upgrade" an entity (normal → champion, boss stage 1 → stage 2), use replace-entity: despawn the current entity and spawn a new one at its position.

**`worldQuery` return shape (per component kind).** All handles expose `id`, `position`, and `tags`. Kind-specific data lives in a `component` sub-object for component kinds that expose their fields (`light`, `emitter`); `transform` handles omit `component` entirely in this plan since rotation and scale are not yet exposed. The `"light"` branch additionally hoists `isDynamic` to the top level — `isDynamic` is a light-specific routing discriminator (baked vs. runtime dynamic), not a general pattern. Other component kinds do not hoist fields unless a similar per-kind routing discriminator exists. The minimum shapes are:

- `"light"`: `{ id, position, tags, isDynamic, component: { ... light fields ... } }`. Top-level `isDynamic` mirrors the nested copy per `scripting.md` §10.
- `"transform"`: `{ id, position, tags }` at minimum. Rotation and scale are not exposed in this plan; the handle is `{ id, position, tags }` only. A future plan may add a `component` field when rotation becomes scriptable. Note: transforms are universal — every live entity has one. `worldQuery({ component: 'transform' })` returns every live entity. There is no soft cap; modders should filter by tag when they need a subset. Document this in Task 4. The `"transform"` branch iterates `EntityRegistry`'s live entity slots directly — since every live entity has a transform, iterating live slots and iterating by `ComponentKind::Transform` are equivalent, but the live-slot walk is the canonical approach.
- `"emitter"`: `{ id, position, tags, component: { ...BillboardEmitterComponent fields... } }`. No field hoisting — no `isDynamic`-equivalent routing discriminator exists for emitters.
- `"particle"`: returns an empty list. Per `entity_model.md` §8, "scripts never observe individual particles." The string is whitelisted only to avoid `ScriptError` on a known component name; the result is always `[]`. Implementation note: the `parse_query_filter` branch for `"particle"` should short-circuit to an empty result rather than walk the registry.
- `"sprite_visual"`: returns an empty list, same rationale and same short-circuit. Sprite visuals are an internal rendering detail of the particle system; scripts have no business iterating them individually.

Handle wrappers follow the per-entity-type vocabulary pattern from `sdk/lib/entities/lights.{ts,luau}`: emitter handles live in `sdk/lib/entities/emitters.{ts,luau}` (already present); create `sdk/lib/entities/transforms.{ts,luau}` exporting a `TransformHandle` type with at minimum `{ id: EntityId, position: Vec3, tags: string[] }`. The `component` field is omitted for now — rotation and scale are not yet exposed through the scripting API. The AC above pins the verifiable observables: every supported string returns without `ScriptError`, and the two no-op kinds return `[]`.

**`worldQuery` filter expansion:**

```typescript
// All of these should work after Task 2:
world.query({ component: "transform" });
world.query({ component: "light" });
world.query({ component: "emitter" });
world.query({ component: "particle" });
world.query({ component: "sprite_visual" });
world.query({ component: "light", tag: "campfire" });
```

**PRL section wire format:** all integers and `f32` values little-endian (IEEE-754 little-endian for `f32`). The section starts with a `u32` `entry_count` header (the number of `MapEntity` records that follow). Fields appear in this order: classname, origin, angles, properties, tags. For each entry: `classname` is a `u32`-length-prefixed UTF-8 string; `origin` and `angles` are three `f32` each (12 bytes each); `properties` is a `u32` count followed by count × (`u32`-length-prefixed key string, `u32`-length-prefixed value string) pairs; `tags` is a `u32` count followed by count × `u32`-length-prefixed UTF-8 strings. This mirrors the `TextureNamesSection` and `LightTagsSection` layout in `crates/level-format/src/` (little-endian, `u32` count header, `u32` length prefix per string, no null terminators). Empty lists serialize as `u32(0)` with no following bytes — the count field is always present.
