# postretro-lighting (CPU-math)

> Epic: `E19--render-stack-decomposition`. Splits `crate::lighting` into a CPU-math crate; the GPU pools move into the renderer crate at `E19--renderer-gpu`.

## Goal

Extract the wgpu-free lighting math (light packing, specular/influence packing) into a CPU-only crate shared by the renderer and gameplay subsystems. Default `cargo check/test -p postretro-lighting` is wgpu-free/VM-free and does not compile GPU deps; downstream renderer crates still rebuild when they consume changed packer APIs.

## Scope

### In scope
- `postretro-lighting`: extract the wgpu-free packer surface from `mod.rs` — `pack_light`, `pack_light_with_slot`, `pack_lights`, `pack_lights_with_slots_into`, `patch_shadow_slots`, `patch_cube_slots`, `light_reaches_visible_cell`, `entity_occluder_eligible`, `GPU_LIGHT_SIZE`, `SHADOW_SLOT_BYTE_OFFSET`, and `CUBE_SLOT_BYTE_OFFSET`. Extract `NO_SHADOW_SLOT` from `spot_shadow.rs`; update the `mod.rs` sentinel call sites to import it from the crate. Do **not** move `mod.rs` wholesale; the current module declares GPU submodules.
- Move `influence.rs`'s `pack_influence` only. `LightInfluence` already lives in `postretro-render-data` (epic Decision 10) and must not be re-added to lighting.
- Move `spec_buffer.rs` (`pack_spec_lights`, `SPEC_LIGHT_SIZE`, `SPEC_LIGHT_TYPE_POINT`, `SPEC_LIGHT_TYPE_SPOT`, `SPEC_LIGHT_TYPE_DIRECTIONAL`, `SPEC_LIGHT_SDF_FLAG`).
- `cone_frustum.rs` does **not** live here — it moved to `postretro-render-data` (`E19--render-data`). It is geometry/AABB math with the widest fan-out (model/weapon/cull/renderer); the shared row-math now lives beside it, and `compute_cull` consumes that lower crate. See `E19--render-data` and the hub Decision.
- The packers' byte-layout constants travel with them (shared with WGSL).
- Re-point the cross-subsystem consumers of the extracted surface — GPU cull/shadow packer imports, current `light_reaches_visible_cell` / `entity_occluder_eligible` consumers, and `crates/postretro/src/scripting/systems/light_bridge.rs` — to the crate. The `cone_frustum`/`Aabb` consumers (weapon hit-zones, model, cull/shadow) now depend on `postretro-render-data`, not `postretro-lighting`.
- Depend by default on `glam`, `postretro-level-loader = { default-features = false }` for the cfg-free `MapLight`, `LightType`, `FalloffModel`, and `ShadowType` types, and `postretro-render-data` (`LightInfluence`). Default packer path has no direct `serde` dep. `script-ffi` enables optional `serde_json`, `postretro-foundation`, `postretro-entities` with `postretro-entities/script-ffi`, and `postretro-scripting-core`. VM crates arrive only through the `script-ffi` feature path, directly or transitively through those dependencies.
- Add an optional `script-ffi` feature gating light-domain script wiring (Task 2), off by default, per the `scripting.md §12` Cargo pattern. When enabled it pulls the VM crates/substrate the wiring needs; default builds stay wgpu/VM-free.
- Move only light-domain scripting pieces from `crates/postretro/src/lighting/script_primitives.rs` into `postretro-lighting` under `script-ffi`: `register_light_entity_primitives`, `register_sequenced_light_primitives`, `collect_light_handles`, `handles_to_json`, `LightQueryHandle`, `LightAnimation`/`LightEntity`/`LightKind`/`FalloffKind`/`LightComponent` type registration, and the light animation validation/apply path. `LightAnimation`, `LightComponent`, and related entity data types remain owned by `postretro-entities`; only light-domain registration, validation, and apply wiring moves to `postretro-lighting` behind `script-ffi`. Moved registrars/helpers must be public exports or re-exports from `postretro-lighting` for external callers in `postretro`; `collect_light_handles` + `handles_to_json` may collapse into one public helper if `LightQueryHandle` stays opaque, but the JSON wire shape must not change. Compatibility shims/wrappers, if needed, remain only as thin `postretro` call-site paths and delegate to `postretro-lighting`.

### Out of scope
- The GPU pool structs `SpotShadowPool`, `CubeShadowPool`, `LightmapResources` (`spot_shadow.rs`/`cube_shadow.rs`/`lightmap.rs`) — they own wgpu and move into `postretro-renderer` at `E19--renderer-gpu`. `NO_SHADOW_SLOT` moves into `postretro-lighting`; `spot_shadow.rs` keeps `crate::lighting::spot_shadow::NO_SHADOW_SLOT` as a compatibility re-export. Other GPU pool modules and renderer code may keep that path, or import `postretro_lighting::NO_SHADOW_SLOT` directly if compilation needs it. Pool/resource behavior stays untouched.
- `lighting/chunk_list.rs` (`ChunkGrid`, `CHUNK_GRID_UNIFORM_SIZE`) — CPU byte-payload planning for renderer buffers. It belongs in `postretro-render-cpu`, not lighting.
- The marshalling substrate `script_primitives.rs` calls — stays in `scripting-core` (the VM-coupled registry/typedef/marshalling substrate).
- Cross-domain world-query/shared type registration stays outside `postretro-lighting`, near `scripting/primitives/mod.rs` and `entity_world_primitives.rs`. This includes `WorldQueryComponent`, `WorldQueryFilter`, `Entity`, `EmitterEntity`, and non-light query dispatch. Carve these out before/while moving the light-domain pieces unless a registration is strictly light-only.

Default features also show no wgpu/winit/glyphon/kira, no `mlua`/`rquickjs`, and no direct `serde` dependency; direct `serde` is allowed only behind `script-ffi` if needed, while explicitly documented transitive serde deps may remain.

They still compile in their current home (they move at `E19--renderer-gpu`).

## Tasks

### Task 1: Extract postretro-lighting CPU-math
Create the crate, extract the wgpu-free packer surface from `mod.rs` and `NO_SHADOW_SLOT` from `spot_shadow.rs`, move `influence.rs`'s `pack_influence`, move `spec_buffer.rs` + its size/flag/type constants, widen boundary symbols, and re-point the GPU cull/shadow packer imports, `light_reaches_visible_cell` / `entity_occluder_eligible` consumers, and `crates/postretro/src/scripting/systems/light_bridge.rs`. Inventory to update: `render/mod.rs`, `renderer_resources.rs`, `renderer_init_resources.rs`, `renderer_light_slots.rs`, `render/renderer_lighting.rs`, `scripting/systems/light_bridge.rs`, `render/tests/shader_tests.rs`, and `render/tests/light_filter_tests.rs`. `spot_shadow.rs` imports/re-exports `NO_SHADOW_SLOT` as a compatibility path; pool structs/resources stay behaviorally untouched. `LightInfluence` stays in `postretro-render-data`; `pack_influence` imports it from there. `cone_frustum.rs` is **not** part of this move — it lives in `postretro-render-data` (`E19--render-data`), and its consumers already re-point there.

### Task 2: Descend light-domain script wiring behind `script-ffi`
**Decision (was open question 1): light-domain scripting wiring descends into `postretro-lighting` behind the optional `script-ffi` feature.** Do not move `lighting/script_primitives.rs` wholesale. Split the current mixed file. Move `register_light_entity_primitives`, `register_sequenced_light_primitives`, `collect_light_handles`, `handles_to_json`, `LightQueryHandle`, the light animation validation/apply path, and the light-only type registrations (`LightAnimation`, `LightEntity`, `LightKind`, `FalloffKind`, `LightComponent`) from `crates/postretro/src/lighting/script_primitives.rs` into `postretro-lighting` under `script-ffi`. Keep cross-domain world-query/shared type registration near `scripting/primitives/mod.rs` / `entity_world_primitives.rs`: `WorldQueryComponent`, `WorldQueryFilter`, `Entity`, `EmitterEntity`, and non-light query dispatch do not become lighting-owned.

The marshalling substrate stays in `scripting-core`; aggregation stays explicit. `postretro_lighting::script_ffi::register_light_types` is the feature-gated public export/re-export for the relocated light-only type registrar. `crates/postretro/src/scripting/primitives/mod.rs::register_all` imports/calls it after the shared base types and before `register_light_entity_primitives`, preserving current typedef order/coverage from `scripting/primitives/mod.rs` + `lighting/script_primitives.rs`; generated typedef output/order remains byte-identical except crate paths/import plumbing. `entity_world_primitives.rs` calls `collect_light_handles` / `handles_to_json` for the light branch of `worldQuery`; `Session::build` owns sequenced registry setup and calls the moved `register_sequenced_light_primitives` path from the crate. `setLightAnimation` is the primitive name; scope is `DefinitionOnly`; params are `id: EntityId`, `animation: LightAnimation | null`. `null` clears. Invalid period, empty channels, and zero/non-finite direction samples error as today. Non-unit directions normalize. Phase normalizes via `rem_euclid`. The sequenced registrar keeps the same primitive name and dispatch/apply behavior, not a new runtime script primitive. `LightAnimation`, `LightComponent`, `LightKind`, and `FalloffKind` remain owned by `postretro-entities`. The `script-ffi` module imports lower-crate types only: `Vec3Lit` from `postretro-foundation`; `ScriptCtx`, `EntityRegistry`, `EntityId`, `Component`, `ComponentKind`, `ScriptError`, `LightComponent`, `LightAnimation`, `LightKind`, and `FalloffKind` from `postretro-entities`; and `PrimitiveRegistry`, `ContextScope`, `SequencedPrimitiveRegistry`, and `SequenceError` from `postretro-scripting-core`. It must not depend on `postretro` or renderer/GPU crates. Direct move is expected; if source drift forces compatibility, keep only thin call-site wrappers in `postretro`; do not introduce callback/trait shims or change ownership without revising the spec. The `postretro` dependency on `postretro-lighting` enables `script-ffi`; default `postretro-lighting` builds keep the light scripting module cfg-disabled. Principle: `scripting.md §12` handler-placement spirit — wiring co-locates with its subsystem when it is subsystem-owned. This is the precedent the Epic 16 combat crate will mirror. With `script-ffi` off, the crate stays VM-free; with it on, VM crates arrive only through the `script-ffi` feature path, directly or transitively through optional dependencies.

Light type registration contract:

| Type | Registered shape |
|---|---|
| `LightKind` | variants `Point`, `Spot`, `Directional` |
| `FalloffKind` | variants `Linear`, `InverseDistance`, `InverseSquared` |
| `LightAnimation` | fields `periodMs: f32`, `phase: Option<f32>`, `playCount: Option<u32>`, `startActive: Option<bool>`, `brightness: Option<Vec<f32>>`, `color: Option<Vec<Vec3>>`, `direction: Option<Vec<Vec3>>` |
| `LightComponent` | fields `origin: Vec3`, `lightType: LightKind`, `intensity: f32`, `color: Vec3`, `falloffModel: FalloffKind`, `falloffRange: f32`, `coneAngleInner: Option<f32>`, `coneAngleOuter: Option<f32>`, `coneDirection: Option<Vec3>`, `isDynamic: bool`, `animation: Option<LightAnimation>`; `animatedSlot` remains serde-visible but omitted from typedef registration |
| `LightEntity` | fields `id: EntityId`, `position: Vec3`, `isDynamic: bool`, `tags: Vec<String>`, `component: LightComponent` |

`setLightAnimation` accepts brightness/color animation for authored static and dynamic lights; there is no `is_dynamic` rejection gate. Finite `phase` values normalize with `rem_euclid(1.0)`; non-finite phase normalizes to `0.0`.

Durable boundary decisions are captured in `context/lib/rendering_pipeline.md` (wgpu-free light packers descend; GPU pools stay renderer-owned) and `context/lib/scripting.md` §4/§12 (light-domain wiring descends behind `script-ffi`; shared world-query/type aggregation stays outside lighting).
