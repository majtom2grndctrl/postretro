# Auto-generate SDK prelude at build time

> **Status:** draft
> **Related source:** `crates/script-compiler/src/main.rs` · `crates/postretro/src/scripting/quickjs.rs` · `crates/postretro/Cargo.toml` · `sdk/lib/index.ts`
> **Related plans:** `context/plans/drafts/scripting-tools-dedup/index.md` (independent — different code path, see Open questions) · `context/plans/drafts/script-driven-light-bake/index.md` (consumer of the prelude — see Open questions)

---

## Goal

Make `cargo build` self-sufficient on a fresh clone. Today the engine `include_str!`s `sdk/lib/prelude.js`, but `*.js` is gitignored and the file is never committed — so a fresh checkout fails to compile until a developer runs an undocumented `scripts-build --prelude` step. After this change, the prelude is generated by a `build.rs` in the engine crate from the committed `sdk/lib/**/*.ts` sources, with no manual step and no PATH dependency.

## Scope

### In scope
- Split `postretro-script-compiler` (binary `scripts-build`) into a library + thin bin wrapper.
- Add `build.rs` to the `postretro` crate that calls the library to generate `prelude.js` into `OUT_DIR`.
- Switch the QuickJS `include_str!` callsite to read from `OUT_DIR`.
- Stop tracking `sdk/lib/prelude.js` in the working tree (remove the file; it is already gitignored).
- Update doc/comment references that point developers at the manual regeneration command.

### Out of scope
- Changing the `scripts-build` CLI surface (`--in/--out`, `--prelude/--sdk-root/--out`). The bin keeps its existing flags.
- Touching the Luau prelude wiring — Luau files are embedded directly from `sdk/lib/**/*.luau` and unaffected.
- Touching the `.d.ts` / `.d.luau` generation pipeline — those artifacts are committed, consumer-facing, and have a separate drift test.
- Folding the `scripts-build` runtime-discovery / mtime helpers into the new library (that is the `scripting-tools-dedup` draft's scope).
- Regenerating the prelude inside `prl-build` for the script-driven light bake (see Open questions).

## Acceptance criteria
- [ ] On a fresh clone with no `sdk/lib/prelude.js` and no `scripts-build` on `PATH`, `cargo build -p postretro` succeeds end-to-end without a manual prelude step.
- [ ] After `cargo build`, the QuickJS prelude evaluated by the engine is byte-identical to what `cargo run -p postretro-script-compiler -- --prelude --sdk-root sdk/lib --out <tmp>` produces against the same tree.
- [ ] Editing any `sdk/lib/**/*.ts` file (entry, re-exported module, or transitive import) and rebuilding causes the prelude to regenerate; editing an unrelated file (e.g. a `.md` in `sdk/`) does not.
- [ ] The `scripts-build --prelude --sdk-root <DIR> --out <OUTPUT.js>` CLI continues to work and produces identical output to the build-script path.
- [ ] `sdk/lib/prelude.js` is absent from `git ls-files` after this change.
- [ ] Existing `cargo test -p postretro` (including `sdk_prelude_installs_globals` in `quickjs.rs` and the typedef drift tests) passes unchanged.
- [ ] No comment, doc, or context file still tells a developer to manually regenerate `sdk/lib/prelude.js` (the regen command is no longer required for normal workflow; the library API is the new entry point).

## Tasks

### Task 1: Library-ize `postretro-script-compiler`
Move the bundler driver out of `crates/script-compiler/src/main.rs` into a new `crates/script-compiler/src/lib.rs`. Expose two functions: `bundle_entry(entry: &Path) -> Result<String>` and `bundle_prelude(sdk_root: &Path) -> Result<String>` (both returning the JS source as a `String`; file-write is the caller's job). Also expose a convenience `write_prelude(sdk_root: &Path, out_path: &Path) -> Result<()>` since both the bin and the build script want the same write-with-mkdir-parent behaviour. Add `[lib] name = "postretro_script_compiler" path = "src/lib.rs"` to `Cargo.toml` alongside the existing `[[bin]]`. Move all swc-touching code (loaders, resolvers, AST visitors) into the library; the bin retains only `main`, `parse_args`, `print_usage`, and the `CliMode` enum, calling into the library for the actual work. Move the `#[cfg(test)] mod tests` block (currently exercising `bundle()`) into `lib.rs` so tests target the library API.

### Task 2: `build.rs` for the engine crate
Add `crates/postretro/build.rs`. It reads `CARGO_MANIFEST_DIR`, walks up to the workspace root, and resolves `sdk/lib`. It calls `postretro_script_compiler::write_prelude(sdk_root, &Path::new(&env::var("OUT_DIR")?).join("prelude.js"))`. It then walks `sdk/lib/` recursively and emits `cargo:rerun-if-changed=<path>` for every `.ts` file found (so adding a new file under `sdk/lib/` causes Cargo to re-run the build script next build) plus one `cargo:rerun-if-changed=sdk/lib` line so directory-mtime changes (file creation / deletion) also retrigger. Add `[build-dependencies] postretro-script-compiler = { path = "../script-compiler" }` to `crates/postretro/Cargo.toml`. The build script must NOT depend on `scripts-build` being installed — it links the library directly.

### Task 3: Switch the engine `include_str!` to `OUT_DIR`
In `crates/postretro/src/scripting/quickjs.rs`, change `const SDK_PRELUDE_JS: &str = include_str!("../../../../sdk/lib/prelude.js");` to `const SDK_PRELUDE_JS: &str = include_str!(concat!(env!("OUT_DIR"), "/prelude.js"));`. Update the doc-comment above it: drop the manual regeneration command (no longer needed) and replace with a one-line note that the prelude is generated by `build.rs` from `sdk/lib/**/*.ts`. The `source_name: "sdk/lib/prelude.js"` string in `evaluate_prelude` stays — it is the human-readable label QuickJS uses in stack traces and is not a path lookup.

### Task 4: Remove the working-tree prelude and update docs
Delete `sdk/lib/prelude.js`. Update `sdk/lib/index.ts`'s header comment that says "regenerate sdk/lib/prelude.js" — replace with a note that the prelude is built automatically. Update `context/lib/scripting.md` §7 ("`sdk/lib/prelude.js` (committed, regenerated when any `sdk/lib/**/*.ts` changes)") and §8 ("Prelude regeneration"): the prelude is no longer committed; the manual `scripts-build --prelude` command becomes an optional debugging tool, not a required step. (Per the planning rules, do not edit `context/lib/` during drafting — flag this update for the implementing pass.) Audit `context/plans/drafts/script-driven-light-bake/index.md` for assumptions about a committed `sdk/lib/prelude.js` on disk (see Open questions); do not edit, just record findings in `research.md` if needed.

## Sequencing

**Phase 1 (sequential):** Task 1 — every other task depends on the library API existing.
**Phase 2 (sequential after Phase 1):** Task 2 — wires build.rs to the new library.
**Phase 3 (concurrent after Phase 2):** Task 3 and Task 4 — independent file edits; both compile against the artifact Task 2 produces.

## Rough sketch

**Library API (`crates/script-compiler/src/lib.rs`):**
```rust
// Proposed surface — finalize during Task 1.
pub fn bundle_entry(entry: &Path) -> anyhow::Result<String>;
pub fn bundle_prelude(sdk_root: &Path) -> anyhow::Result<String>;
pub fn write_prelude(sdk_root: &Path, out_path: &Path) -> anyhow::Result<()>;
```
`bundle_prelude` resolves `<sdk_root>/index.ts`, canonicalizes it, and calls the existing `bundle(entry, prelude=true)` internal helper. `write_prelude` wraps that with the parent-dir-create + `fs::write` logic currently inlined in `main.rs::run`.

**Build script (`crates/postretro/build.rs`):**
- Resolve `sdk_root` as `<CARGO_MANIFEST_DIR>/../../sdk/lib` (workspace layout: `crates/postretro/` → `../..` → repo root).
- Call `postretro_script_compiler::write_prelude(&sdk_root, &out_dir.join("prelude.js"))`. Panic on `Err` so the engine build fails loudly, not with a silent stale prelude.
- Walk `sdk_root` once with `std::fs::read_dir` recursion (no extra deps), filtering to `*.ts`, and emit `cargo:rerun-if-changed=` for each. Emit one for the directory itself to catch additions.
- Emit `cargo:rerun-if-changed=build.rs` (Cargo does this implicitly but explicit is clearer).

**Engine include site (`crates/postretro/src/scripting/quickjs.rs`):**
```rust
// Proposed.
const SDK_PRELUDE_JS: &str = include_str!(concat!(env!("OUT_DIR"), "/prelude.js"));
```
`OUT_DIR` is a Cargo-injected env var visible to `include_str!` via `concat!(env!(...), ...)`. This is the established pattern for build-script-generated source.

**Workspace dependency graph after the change:**
- `postretro` (lib + bin) → `[dependencies] postretro-level-format` (unchanged) + `[build-dependencies] postretro-script-compiler` (new edge).
- `postretro-script-compiler` (lib + bin `scripts-build`) → swc deps (unchanged).
- No cycle: `script-compiler` does not depend on `postretro`.

## Open questions

1. **swc build-time cost — watch item.** swc is already a workspace dependency, so this plan does not add new cold-compile cost today. If cold build time becomes a problem in practice, the fallback is listing `bun` as an explicit system requirement for engine devs and modders using the TS API, and invoking it as a subprocess instead. Not actionable now; revisit if complaints surface.
2. ~~**`prl-build` (level compiler) prelude usage.**~~ Resolved. `prl-build` will mirror the engine: `postretro-script-compiler` added as `[build-dependencies]`, prelude embedded via its own `build.rs` using the same `write_prelude` call. Deterministic, no PATH lookup, same pattern as the engine. A lighter TS processing crate is an option if swc ever becomes a burden, but not a priority. The `script-driven-light-bake` plan should follow this approach.
3. **Relationship to `scripting-tools-dedup`.** That draft factors `scripts-build` *runtime discovery* and *mtime freshness* helpers into a shared crate (`postretro-scripts-tools`) consumed by the engine's hot-reload watcher and `prl-build`'s subprocess invocation. That is a different code path from the *bundler library* this plan extracts — runtime subprocess plumbing vs. compile-time bundler API. The two plans are independent and can land in either order. If both ship, the workspace ends up with `postretro-script-compiler` (bundler lib + bin) and `postretro-scripts-tools` (runtime helpers) as siblings; consider whether they should merge under a single `postretro-scripts-tools` crate after both ship — leaving as a follow-up.
4. ~~**Re-run granularity.**~~ Resolved. Directory mtime is unreliable only on NFS and some Docker overlay setups — not relevant here. APFS, ext4, and NTFS all update directory mtime on add/delete. Emit per-file `rerun-if-changed` plus one for the directory; no fallback needed.
5. **`scripts-build` bin still ships.** The bin remains for one-off CLI use (debugging the prelude output, regenerating on demand for inspection, future `prl-build` integration per Open question 2). No deprecation; the bin and lib coexist.
6. **Workspace-root resolution in `build.rs`.** Hard-coding `../../sdk/lib` from `CARGO_MANIFEST_DIR` works because the workspace layout is fixed (`crates/postretro/` is always two levels under the repo root). If we ever move crates around, the build script breaks loudly at compile time — acceptable, since any such move would already require touching `build.rs`. Alternative (env var passing the workspace root) adds ceremony for no current benefit.
