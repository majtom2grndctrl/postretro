# Gravity Primitives (M7)

> **Status:** ready
> **Depends on:** scripting-primitives-folder refactor — ready
> **Prerequisite for:** Movement Scripts (M7)
> **Related:** `context/lib/scripting.md` · `context/lib/build_pipeline.md`

---

## Goal

Expose world gravity to scripts via `world.getGravity()` and `world.setGravity()`, with the starting value set per-map via an `initialGravity` worldspawn KVP.

**Sign convention:** negative-down. Standard Earth gravity is `-9.81`. This matches the existing `WORLD_GRAVITY = -9.81` constant in `particle_sim.rs` — no internal conversion is needed. A positive value would cause upward acceleration.

---

## Tasks

### 1. `initialGravity` worldspawn KVP and PRL field

- Add `initialGravity` to the `worldspawn` entity definition in `sdk/TrenchBroom/postretro.fgd`. Type: `float`, units: m/s², sign convention: negative-down (`-9.81` = standard gravity). No default value. Description: starting gravity; negative = downward, positive = upward; may be changed at runtime via `world.setGravity()`.
- In `prl-build` (`postretro-level-compiler`): read `initialGravity` from worldspawn KVPs. Error and halt at compile time if absent. Emit the value as a new `f32` field on the existing worldspawn/header PRL section (following the same pattern as `fog_pixel_scale`).
- In the engine (`prl::load_prl`): read `initial_gravity: f32` from the PRL field; store it on `LevelWorld`.
- Add a `gravity: Rc<Cell<f32>>` field to `ScriptCtx` (following the documented "extend by adding one field per subsystem" pattern). `ScriptCtx` is the sole owner; `App` reads and writes through `self.script_ctx.gravity` directly. Seeded from `level.initial_gravity` in the `!level_load_fired` cold-path branch before any script runs; that branch is the single seed site and is reused on future level reload.
- `ScriptCtx::new` initializes gravity to `f32::NAN` so any code path that bypasses `prl::load_prl` surfaces immediately in `particle_sim::tick`. The `worldSetGravity` primitive rejects non-finite writes, so scripts cannot reintroduce the sentinel.

### 2. Gravity primitives

Move `worldQuery` from `scripting/primitives/mod.rs` to the new `scripting/primitives/world.rs` in this same change. Register the gravity primitives in `world.rs` alongside it (`ContextScope::Both`):

- Flat primitive name `worldGetGravity` → `world.getGravity() → number` in the SDK. Returns `current_gravity` via `ScriptCtx::gravity`.
- Flat primitive name `worldSetGravity` → `world.setGravity(value: number)` in the SDK. Writes `value` to `ScriptCtx::gravity`. Rejects NaN and non-finite inputs with `log::warn!` and ignores the call. Effect is immediate and persists until the next level load or another `setGravity` call.

The naming pattern mirrors `worldQuery` → `world.query()` and leaves `area.setGravity()` as a natural future parallel.

Wire `particle_sim.rs` to read `current_gravity` from `ScriptCtx` (via the same `Rc<Cell<f32>>` handle) instead of the hardcoded `WORLD_GRAVITY` constant. Remove the constant.

### 3. SDK update + documentation

- Update `sdk/lib/world.{ts,luau}` to add `getGravity` and `setGravity` as hand-authored methods on the `world` SDK object. Each method delegates to the corresponding flat primitive (`worldGetGravity` / `worldSetGravity`). These are hand-authored — not generated by `gen-script-types`.
- Run `cargo run -p postretro --bin gen-script-types` to regenerate `sdk/types/postretro.d.ts` and `sdk/types/postretro.d.luau`. This picks up the new flat primitive names; commit the updated files so the drift-detection test passes (`cargo test`). The drift test validates primitive registry output only — it does not validate SDK method shapes.
- Add a **World** section to `docs/scripting-reference.md` (or extend it if already present) covering both methods: signatures, units, sign convention (`-9.81` = standard gravity), and a cross-reference to the `initialGravity` worldspawn KVP.

## Acceptance criteria

- `world.getGravity()` returns the `initialGravity` value from the PRL at level load.
- `world.setGravity(v)` updates the value returned by subsequent `world.getGravity()` calls.
- Particle buoyancy is affected by `setGravity` — `particle_sim.rs` reads from `current_gravity`, not the former constant.
- Engine (PRL load) rejects a PRL built without `initialGravity`; `prl-build` rejects a `.map` missing `initialGravity` from worldspawn.
- Loading a second level resets gravity to that level's `initialGravity`.
- Both QuickJS and Luau paths return the same value via `world.getGravity()`.
- `world.setGravity(NaN)` and `world.setGravity(Infinity)` are silently ignored (warning logged, value unchanged).
- Both methods documented in `docs/scripting-reference.md` with sign convention stated.
- SDK drift-detection test passes (`cargo test`).
