# Perf ‚Äî Animated SH Light Culling

## Goal

Cut the per-frame SH compose workload and the GPU storage footprint of baked animated-light delta volumes so a single bind group fits inside the WebGPU `max_storage_buffer_binding_size` spec floor (128 MiB) and the per-probe inner loop touches only the lights that actually reach the probe. Today, `campaign-test.prl` ships 21 dense per-light AABB grids that total ~185 MB after f16‚Üíf32 expansion, and every base SH probe loops over every animated light. The raised wgpu limit (512 MiB) just landed as a stopgap; this plan removes that load-bearing dependency.

This spec is an experiment. Build it, run `campaign-test` end-to-end, and measure: does the sparse representation alone reach a good quality/frame-time balance, is further perf work warranted, or are debug-tool tuning knobs enough? The debug tool is the sandbox for any knob introduced here ‚Äî promotion to a user-facing quality setting waits on the global quality-settings spec after the UI system lands. The ACs below gate that the experiment runs honestly (correct bake, clean boot, reject path); the quality and perf numbers are measured findings that feed the decision, not contracts.

## Scope

### In scope
- Sparse per-light delta storage on disk: do not bake or ship probes outside the light's portal-reachable region within its AABB.
- f16 probes stay f16 through to the shader (drop the f16‚Üíf32 expansion in `build_delta_buffers`).
- Coarse spatial index over the base SH volume that maps each affinity cell (one compute workgroup) to a short list of overlapping animated lights.
- Compose shader iterates only the indexed light list per affinity cell (replaces the current "loop all lights, AABB test").
- New PRL section version bump for the sparse + index payload (pre-release; no compat shim).
- `campaign-test.prl` rebakes end-to-end and renders without raising any wgpu limit past spec floors.
- A debug-tool slider that scales the animated-light SH delta contribution at runtime ‚Äî the experimentation knob for judging the quality/frame-time tradeoff. Sandboxed in the existing dev-tools sliders; a candidate for the future global quality-settings spec, not a user-facing setting yet.

### Out of scope
- Octree / BVH spatial index. Coarse uniform grid is the M10 minimum. The per-cell light loop is shared across the workgroup's 64 threads (read once per workgroup), so a long cell list is a longer shared loop, not per-thread cost; there is no per-thread budget to overflow.
- Dynamic re-binning when lights move. Animated lights are static-positioned in the bake; movement is intensity/color only.
- The 1,962,545-tile compute-dispatch issue on the animated-lightmap install path. Same dense-everything root cause but a different code path; tracked separately. Note in the implementation: if the affinity-cell decomposition introduced here is reusable for that pass, mention it in the commit message so the follow-up plan can pick it up.
- Lowering `REQUIRED_STORAGE_BUFFER_BINDING_SIZE`. Leave the raised limit in place; this plan makes it non-load-bearing, not removed.
- Tighter per-light AABBs. The reachable-region clip subsumes most of the win.

## Acceptance criteria

- [ ] `cargo run -p postretro-level-compiler -- content/dev/maps/campaign-test.map -o /tmp/out.prl` succeeds; resulting `.prl` is smaller than the pre-change baseline for the `DeltaShVolumes` section.
- [ ] Sparse bake drops out-of-region cells: for at least one animated light on `campaign-test.prl`, the emitted probe count (reached affinity cells √ó `AFFINITY_FACTOR¬≥`) is strictly less than its full-AABB probe count. (Emission is dense by cell, so this is a cell-granular drop, not a per-probe drop.) Log both the emitted-cell probe count and the full-AABB probe count at bake time.
- [ ] `cargo run -p postretro -- /tmp/out.prl` boots and reaches the first rendered frame with no wgpu validation errors.
- [ ] Footprint is measured and reported (not contracted): a startup log line reports per-binding byte sizes for every storage buffer binding in the `SH Compose Bind Group` (`@group(1)` ‚Äî the delta payload buffers plus the two new CSR buffers `affinity_offsets`/`affinity_lights`) plus the combined total. The target is below 128 MiB on `campaign-test.prl`; if the sparse form lands over, that is a recorded experiment finding feeding the Task 5 decision (occupancy-bitmap fork vs. debug-knob), not a silent failure.
- [ ] Compose shader's per-probe loop bound is the affinity-cell light-list length, not `delta_light_count`. Verifiable by reading `sh_compose.wgsl` post-change.
- [ ] No visual regression in animated-lit regions on `campaign-test.prl` (manual-visual): A/B the dev-tools full-compose view before and after this change ‚Äî animated-lit areas show no objectionable artifact, banding, or popping. An eyeball check, not a numeric gate; exact parity to the dense bake is explicitly not the goal. The reachable-region clip only drops near-zero out-of-region contributions, and f16 storage is not a precision change (f16‚Üíf32 widening is exact; in-shader `unpack2x16float` is bit-identical to the old CPU-expanded f32). The one genuine sampling change is the removal of trilinear interpolation: v1 baked each light's delta grid at a per-light AABB origin (`light.origin ‚àí range`, independent of the base grid origin), so probe positions fell at arbitrary fractional offsets and the compose shader trilinearly blended 8 delta probes per base probe. The new sub-blocks are coincident with base probes 1:1, so the per-thread read is a point lookup at base-probe positions, replacing interpolated values with base-probe-aligned ones. Justified because delta contributions vary slowly at 1.0 m spacing, but it is a real change in sampled value ‚Äî the manual A/B must specifically watch animated-lit gradients for banding or position shift, not just brightness parity. The A/B must be run on the normal render path with DDGI probe depth-visibility weighting active ‚Äî not in any DDGI-bypass or visibility-off debug mode ‚Äî because the composed delta SH is sampled through that weighting and a bypass would compare values the engine never actually shows.
- [ ] PRL section version reject path: loading a pre-bump `.prl` with the new engine logs a clear "recompile with current prl-build" error and exits, mirroring the existing `DELTA_SH_VOLUMES_VERSION` mismatch path.
- [ ] Spec-floor load is verified: temporarily request `max_storage_buffer_binding_size` at only the WebGPU spec floor (`134_217_728`) and confirm `campaign-test.prl` loads. (The raised limit stays requested as headroom in normal builds.) Coupled to the footprint finding above ‚Äî if the sparse form is over floor this surfaces the same recorded result, not a hard gate.

## Tasks

### Task 1: Affinity grid decomposition (bake)
Produce, per animated light, the set of affinity cells the light's contribution should be stored at. An affinity cell is a coarse cube of base SH probes: 4√ó4√ó4 base probes per cell. `AFFINITY_FACTOR` is locked to `4` ‚Äî it is an invariant, not a tuning knob, because the affinity cell must equal one compose workgroup so the per-cell light list is read once per workgroup and shared across its 64 threads. Changing the factor means changing the compose `@workgroup_size` as a paired edit. The factor still ships in the wire-format `affinity_factor` field so the loader validates it matches the shader's compiled-in size. For each light, intersect the light's AABB with the BSP-portal-reachable region from the light's position. There is no extracted "reachable set from seed leaf" helper; reuse the inline per-light portal flood-fill pattern already in `chunk_light_list_bake.rs:98‚Äì135` ‚Äî seed with `partition::find_leaf_for_point(tree, light.origin)` (`partition/bsp.rs:17`) and BFS the `Portal` adjacency graph (`portals.rs` `Portal { front_leaf, back_leaf }`, fed by `generate_portals`), bypassing the filter when the seed lands in a solid/exterior leaf as that code does. Task 1's deliverable is `per_light_cells: Vec<Vec<u32>>` (per animated light, the list of affinity-cell linear indices it reaches, x-fastest `idx = x + y*dims.x + z*dims.x*dims.y`); Tasks 2 and 3 both consume it. This lives in `crates/level-compiler/` next to the existing delta SH bake.

### Task 2: Sparse delta probe bake
Rework the delta bake so each light's emitted probes correspond only to base-SH-grid positions inside its affinity-cell list (Task 1). The bake stops producing AABB-dense grids and instead emits, per CSR entry, one dense 64-probe (`AFFINITY_FACTOR¬≥`) sub-block of f16 SH coeffs into `delta_subblocks`, parallel to `affinity_lights` at a 28-half per-probe stride (27 coeffs + 1 zero pad), probes ordered x-fastest within the cell ‚Äî in-cell linear index `local = lx + ly*AFFINITY_FACTOR + lz*AFFINITY_FACTOR¬≤` (= `lx + ly*4 + lz*16`), which equals WGSL `local_invocation_index` for `@workgroup_size(4,4,4)`; the bake must write sub-block probes in this exact order (storage form decided ‚Äî see Rough sketch and Wire format). Probes outside the light's reachable region but inside an included cell are written zero (the sub-block is dense by cell). Cells the light does not reach are not emitted at all. `DeltaShProbe` stays `[u16; 27]` in memory (`PROBE_F16_COUNT = 27`, `delta_sh_volumes.rs:27`); the pad to `PROBE_F16_STRIDE = 28` halves is applied at serialize time only (one zero half written per probe), so `PROBE_BYTES` 54‚Üí56 is a wire/buffer stride change, not a CPU-struct width change. The bake logs, per animated light, the emitted-cell probe count (reached cells √ó `AFFINITY_FACTOR¬≥`) and the full-AABB probe count ‚Äî the two numbers AC #2 checks. Updates `DeltaShVolumesSection` to the new shape; bumps `DELTA_SH_VOLUMES_VERSION`.

### Task 3: Per-probe affinity index
Build the runtime lookup: for each affinity cell, the list of animated-light indices whose sparse data covers it. Emit as a flat CSR-style pair: `affinity_offsets: Vec<u32>` (length affinity_cell_count + 1) and `affinity_lights: Vec<u32>` (flat indices). Built at compile time from Task 1's per-light affinity-cell lists. Stored in-section, alongside the sparse probes in the bumped `DeltaShVolumesSection` (committed ‚Äî keeps the load atomic). Loader uploads both as storage buffers in the same group as the existing delta bindings. Loader rejects a `.prl` whose `affinity_factor != AFFINITY_FACTOR` or whose `affinity_dims != ceil(base_dims / AFFINITY_FACTOR)`, and errors clearly if the base ShVolume section (id 20) is absent ‚Äî mirroring the version-mismatch reject path. Emit the startup log line required by the footprint AC: per-binding byte sizes for every storage binding in the SH Compose Bind Group.

### Task 4: Compose shader rewrite
Replace the per-probe `for li in 0..delta_light_count` loop in `sh_compose.wgsl` with: take the affinity-cell index from `workgroup_id` (free ‚Äî `workgroup_id` IS the cell coordinate, since the workgroup grid equals `affinity_dims`; do not recompute it from `base_probe_index`) ‚Üí load `[start, end) = affinity_offsets[cell..cell+1]` ‚Üí loop `for i in start..end` reading `affinity_lights[i]` as the light index. Each iteration reads that light's probe contribution at probe slot `i * 64 + local_invocation_index` (the x-fastest linear index of this thread within the 4√ó4√ó4 workgroup selects the in-cell probe ‚Äî must match the bake's in-cell order, Task 2), scaled by the 28-half per-probe stride into `delta_subblocks`. This is a direct point read, not an interpolated sample: the sub-block probe at this slot is coincident with this thread's base probe 1:1 (the sub-block is the 4√ó4√ó4 base-probe block of the affinity cell, so sub-block local index = base-probe local index in the cell), so there is no inter-probe blend. Drop the old `sample_delta_trilinear` 8-corner read path entirely ‚Äî it existed only because the v1 delta grids had per-light AABB origins independent of the base grid; the new sub-blocks share the base grid's probe positions, so no interpolation is needed. No AABB test, no search, no descriptor-out-of-range path ‚Äî descriptor validation now happens once at bake/load. f16 stays f16 in the storage buffer; sample via `unpack2x16float` (net-new ‚Äî there is no existing `unpack2x16float` use in the WGSL to mirror). Loop bound comes from `affinity_offsets`, not from any light count, so the old `delta_light_count` uniform field has no consumer. Replace it: change the 4th field of `GridDims` from `delta_light_count: u32` to `delta_scale: f32` in both the WGSL `struct GridDims` and the CPU-side struct (`sh_compose.rs:46`). Same slot count (one u32 ‚Üí one f32), so the uniform size and the offset are unchanged ‚Äî the field still occupies byte offset 12, and the CPU upload writes an f32 there instead of a u32 (the override `write_buffer` at `sh_compose.rs:259` writes `f32::to_ne_bytes` now). The shader multiplies the accumulated delta SH by `delta_scale` before adding it to the base bands (i.e. before writing the composed output).

Unify the base-only override and the experimentation knob into this one field: the `dev-tools`-cfg-gated `dispatch` path no longer writes `0` to a light count ‚Äî it writes the slider value to `delta_scale`, where `0.0` IS the base-only override (pure base‚Üítotal copy) and `1.0` is full delta. The debug-tool slider drives `delta_scale` continuously over `0..1` for judging the quality/frame-time tradeoff by eye; sandbox only, not a user-facing setting. The empty-delta case (no animated lights) is handled by the affinity-offset length, not by `delta_scale`.

### Task 5: Run, measure, decide
Run campaign-test end-to-end. Measure: bake time delta, `.prl` size delta, GPU storage footprint of the SH Compose Bind Group, per-frame compose dispatch time (with `POSTRETRO_GPU_TIMING=1`), and within-cell occupancy (the storage driver ‚Äî see Open questions; `AFFINITY_FACTOR` is fixed at `4`, locked to the workgroup size, so it is not a tuning output). Confirm the correctness AC items (clean boot, version reject, out-of-region drop, loop bound). For the spec-floor AC, temporarily set `REQUIRED_STORAGE_BUFFER_BINDING_SIZE` (currently `512 MiB` at `crates/postretro/src/render/mod.rs:916`) to the WebGPU spec floor `134_217_728` and confirm `campaign-test.prl` loads; revert after measuring (the raised limit stays requested in normal builds). Measure the worst-case affinity-cell light-list length and record the ceiling: a per-workgroup shared-loop bound, not a per-thread cost. Confirm the `AFFINITY_FACTOR` constant carries a rationale comment tying it to the workgroup size. If the affinity-cell decomposition is reusable for the animated-lightmap install dispatch (the out-of-scope 1,962,545-tile issue), note that in the commit message.

This task's deliverable is the experiment's findings note: actual footprint vs. the 128 MiB target, compose dispatch time before/after, within-cell occupancy, and a recommendation ‚Äî does the sparse form suffice as-is, is a perf optimization warranted (occupancy bitmap, light cap), or are the debug-tool knobs enough to defer that to the global quality-settings spec? Use the debug-slider knob from Task 4 to characterize the quality/frame-time tradeoff while deciding.

## Rough sketch

Naming: introduce `AffinityCell` as the spatial unit in both bake and runtime. An affinity grid covers the base SH volume's AABB at factor `AFFINITY_FACTOR = 4` (locked to the compose workgroup size; one affinity cell = 4√ó4√ó4 base probes ‚âà 64 m¬≥ at 1.0 m base probe spacing). Affinity-cell index = `floor(base_probe_index_xyz / AFFINITY_FACTOR)`, linearized x-fastest: `idx = x + y*dims.x + z*dims.x*dims.y` ‚Äî the same order the base ShVolume uses (`sh_volume.rs`) and the existing compose WGSL strides `(1, dim.x, dim.x*dim.y)`. ("z-major" elsewhere in the codebase is a label for this x-fastest order, not z-fastest.)

Reachable-region clip (Task 1) does not require new portal/BSP plumbing ‚Äî the compiler already produces portal connectivity for runtime visibility. Reuse the same per-leaf reachability with the light's containing leaf as the seed.

Sparse delta storage (Task 2): committed to the sub-block form. Each light's per-cell contribution is a dense `AFFINITY_FACTOR¬≥` (= 64) probe sub-block of f16 SH coeffs, stored in an array laid out parallel to the CSR `affinity_lights` array ‚Äî entry `i` of `affinity_lights` names the light, sub-block `i` holds that light's 64-probe data for that cell. The same CSR index `i` addresses both; there is no per-light header and no search. The pair form (`(base_idx, coeffs)`) is rejected: it needs a per-probe search to find a probe's contribution, which fights the workgroup dispatch where `workgroup_id` already is the cell and the linear thread index already is the in-cell probe.

Compose shader (Task 4) keeps the existing `@workgroup_size(4, 4, 4)` dispatch over the base SH grid (one thread per base probe). The workgroup grid is `ceil(base_dims / 4)`, which equals `affinity_dims` exactly because `AFFINITY_FACTOR = 4`. So `workgroup_id` IS the affinity-cell coordinate (free, not computed), one workgroup maps 1:1 to one affinity cell, and the per-cell light loop is read once per workgroup and shared across its 64 threads. The whole design anchors here: the affinity cell is the workgroup.

The Animated SH delta volumes paragraph in `context/lib/rendering_pipeline.md ¬ß4` documents the current dense bake. Update at promotion, not in this draft.

## Wire format

`DeltaShVolumesSection` payload version byte (`DELTA_SH_VOLUMES_VERSION`, currently `1`) bumps to `2` ‚Äî this is the payload's first byte, not the enclosing `SectionBlob` version. Layout below; little-endian throughout; integer signedness and widths follow v1.

| Field | Width | Notes |
|---|---|---|
| `version` | `u8` | `= 2`. v1 path rejects with the existing version-mismatch error. |
| `affinity_factor` | `u8` | Cell size in base probes per axis. `= 4`, locked to the compose `@workgroup_size`. Stored so the loader rejects a `.prl` whose factor disagrees with the shader's compiled-in size, not because it varies at bake time. |
| `affinity_dims` | `3 √ó u32` | Affinity grid dimensions = `ceil(base_dims / affinity_factor)`. `base_dims` is read from the base ShVolume section (id 20); DeltaShVolumes load depends on it. |
| `animated_light_count` | `u32` | As v1. |
| `animation_descriptor_indices` | `u32 √ó animated_light_count` | As v1. |
| `affinity_offsets` | `u32 √ó (affinity_cell_count + 1)` | CSR offsets into `affinity_lights`. Last entry = total list length. |
| `affinity_lights` | `u32 √ó affinity_offsets[-1]` | Flat light-index list. |
| `delta_subblocks` | `f16 √ó affinity_offsets[-1] √ó 64 √ó 28` | One dense 64-probe sub-block per CSR entry, parallel to `affinity_lights`; sub-block `i` is the contribution of light `affinity_lights[i]` in that cell. Each probe carries `PROBE_F16_COUNT = 27` SH f16 coeffs (logical count, unchanged). f16 coeffs are read via `unpack2x16float` (2 halves per `u32`); 27 is odd, so each probe's 27 coeffs pad to a padded stride `PROBE_F16_STRIDE = 28` halves (= 56 bytes = 14 `u32`) for clean pair-unpacking. The padding half is written zero and ignored. `PROBE_F16_COUNT` stays 27 as the logical coeff count; `PROBE_F16_STRIDE` is the new on-disk/in-buffer per-probe stride (the existing `PROBE_BYTES = 54` becomes 56). The `delta_subblocks` buffer begins u32-aligned; within each `u32`, coeff `2k` occupies the low 16 bits and coeff `2k+1` the high 16 bits, matching `unpack2x16float` returning `(low, high)`. In-cell probe order is x-fastest (x, then y, then z), matching WGSL `local_invocation_index`; the bake must write sub-block probes in this order so the shader's linear thread index selects the correct in-cell probe. |

Linearization: affinity-cell index and base-probe index both follow the base ShVolume grid's existing x-fastest order ‚Äî `idx = x + y*dims.x + z*dims.x*dims.y` (this is what the codebase calls "z-major"; x varies fastest). `affinity_cell_count = affinity_dims.x * affinity_dims.y * affinity_dims.z`.

Empty animated-light count: `affinity_offsets` is `[0; affinity_cell_count + 1]`, `affinity_lights` is empty. No probe payload.
Sentinel for "no animation descriptor": same `u32::MAX` convention as v1.

## Open questions

- Within-cell occupancy on campaign-test: dense 64-probe sub-blocks waste storage on in-wall / out-of-region probes inside an included cell. For room-shaped reachable regions occupancy is high and waste is second-order; a light reaching down a long thin corridor produces mostly-empty included cells. Measure within-cell occupancy in Task 5. If dense sub-blocks exceed the 128 MiB budget, the fork is an occupancy bitmap with variable per-cell stride (costing a per-cell prefix-sum in the shader) ‚Äî NOT the pair form, which reintroduces the per-probe search.
- The current code has a known correctness bug ‚Äî `grid_dimensions` declared probe count can disagree with the written probe count, causing OOB reads ("likely flicker source" per the warning at `sh_compose.rs:348‚Äì357`). This plan removes the dense grid entirely, which dissolves that bug class. The new shape's analog invariants are (a) `affinity_offsets.len() == affinity_cell_count + 1` and (b) every `affinity_lights[i] < animated_light_count`; both are bake-time asserts re-validated at load. Confirm at Task 4 that these replace the old declared-vs-written mismatch class.
